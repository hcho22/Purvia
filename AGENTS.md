## Two trust models for conversation-shaped tables (do not merge)

There are two parallel chat-message table pairs, on purpose, with **different RLS boundaries**:

- `threads` / `messages` (`supabase/migrations/20260416120000_init_threads_messages.sql`) — the
  knowledge-assistant surface. **Owner-only**: `auth.uid() = user_id`, child delegates to its thread's owner.
  This predicate is the leak-proof core the E4/E6 evals pin; it must stay untouched.
- `conversations` / `conversation_messages` (`supabase/migrations/20260623120000_init_conversations.sql`) —
  the support-widget surface (Epic E). **Workspace-membership**: the ADR-0002 EXISTS-against-
  `workspace_membership` clause (`wm.workspace_id = … and wm.user_id = auth.uid()`), child delegates to the
  parent conversation's workspace. Any member of the workspace can read/claim the queue.

Do NOT collapse these into one table with a `kind` discriminator branching the policy — that puts two
trust models in one predicate (PRD Risk #3 / ADR-0004 reject it). `workspace_membership.role` is
administrative-only (ADR-0002) and must never enter any visibility predicate on these tables.
Cross-workspace zero-leak is pinned by `backend/test_us066_conversations_rls.py` (run via
`python -m backend.test_us066_conversations_rls` against a local Supabase with `DATABASE_URL` set).

## Conversation status machine + derivable deflection (US-067, ADR-0004)

`conversations.status` is a one-way latch `active -> escalated -> resolved` enforced in the DB, not just the service layer.
The `20260623130000_conversation_status_machine.sql` migration installs:

- a CHECK constraint pinning `status in ('active','escalated','resolved')` (added idempotently so it coexists with whatever US-066 defines - exactly once), and
- an idempotent (`create or replace`) `BEFORE INSERT OR UPDATE` trigger (`conversations_status_guard` / `public._conversations_status_guard`) that rejects `escalated -> active` and any `resolved -> *` (resolved is terminal; `resolved -> resolved` no-ops are allowed).

`escalated_at` is a **set-once latch owned entirely by the trigger on both the INSERT and UPDATE paths** - callers never set it. A row born `escalated` is latched `now()` at insert (any other birth status, including `resolved`, gets a null latch); otherwise it is stamped `now()` on the first transition into `escalated` and is preserved verbatim on every later write. An un-latched row always ignores any caller-supplied value, so neither a stray INSERT nor a later UPDATE can plant a timestamp the metric would misread. This is what makes deflection *derivable* instead of a stored `resolved_by_bot`/`resolved_by_human` flag:

- `resolved AND escalated_at IS NULL`     => deflected (bot handled it alone)
- `resolved AND escalated_at IS NOT NULL` => human-handled

Deflection rate from production data (divide-by-zero-guarded):

```sql
select
  count(*) filter (where status = 'resolved' and escalated_at is null)::numeric
  / nullif(count(*) filter (where status = 'resolved'), 0) as deflection_rate
from public.conversations;
```

The runtime "first escalating turn stops the bot pipeline" behaviour is wired separately in US-080; this migration is only the DB-level latch it relies on.
Test: `python -m backend.test_conversation_status_machine` (DB-level, asyncpg as `postgres`; skips cleanly when the local DB / `conversations` table is absent).

## Self-signed Supabase-compatible JWT minting (US-068, ADR-0008)

`backend/supabase_jwt.py:mint_supabase_jwt(sub, ttl_seconds)` is the **single** place the backend issues a Supabase-shaped identity token.
It self-signs a short-lived HS256 JWT with `SUPABASE_JWT_SECRET` (the *same* secret GoTrue signs with), claims `sub` / `role='authenticated'` / `aud='authenticated'` / `iat` / `exp = iat + ttl`, so the token is - to PostgREST and every RLS predicate - indistinguishable from a GoTrue-issued one (`auth.uid()` resolves to `sub`).
This is a new **issuer** beside GoTrue, **not** a new enforcement path - the membership/ACL boundary in the DB is untouched.
Chosen over a GoTrue admin-API session per request (avoids a per-turn round-trip and keeps the service-role key out of the request hot path).

Its only caller is the support bot (US-070): each customer turn mints a ~60s token for `sub = bot_user_id`, calls `match_chunks` as that principal, and discards it.

- `SUPABASE_JWT_SECRET` is a **NEW** env and a **NEW signing surface** (P5 threat-model line): before US-068 the backend held only the anon key (public, non-signing). Whoever holds this secret can forge any identity - server-side only, never embedded client-side. It is **optional**: required only when support is enabled, so the minting helper reads it fail-closed at call time (a knowledge-assistant-only deployment may leave it unset; `main.py` documents it but does not gate startup on it).
- The minted token is **server-side only** and must NEVER reach an HTTP response body, SSE event, or log line bound for the iframe/client.
- The helper mints ONLY for a server-resolved `bot_user_id`, never for a customer- or request-supplied `sub`. PyJWT is now a direct runtime dep (pinned in `requirements.txt`; previously transitive via `supabase`/gotrue).

Test: `python -m backend.test_supabase_jwt` - a unit layer (always runs, no DB/secrets: claim shape, signature-bound-to-secret, TTL validation, fail-closed on missing secret, strict-verifier expiry) plus an integration layer (skips cleanly without local Supabase: a minted token is accepted by PostgREST and resolves `auth.uid()` to `sub`, a token for a different `sub` reads 0 rows, an expired token is rejected with PGRST303).
Note: PostgREST applies a ~30s clock-skew tolerance on `exp`, so the integration expiry check hand-rolls a long-past-`exp` token rather than waiting out a short-TTL minted one.

## Lazy support-bot provisioning + `is_bot` flag (US-069, ADR-0008)

The per-workspace support bot is **not a new content role**. It is an ordinary `auth.users` row plus an ordinary `workspace_membership(role='member')` row, distinguished only by a new boolean flag `is_bot` (`20260624120000_workspace_membership_is_bot.sql`).

- **`is_bot` is a FLAG, not a role.** Like `workspace_membership.role`, it is administrative metadata and must NEVER enter any visibility/retrieval predicate - not `match_chunks`, not `keyword_search`, not the chunks/documents or conversations RLS. The bot is `role='member'`, the administrative-only role grants no content access (ADR-0002), so the bot sees ONLY documents shared to it via `chunk_acl` (share-to-bot), resolved from `auth.uid()` like any other principal. There is **no dedicated `bot` content-role anywhere**. Adding `is_bot` changes the boundary by exactly zero predicates.
- **Exactly one bot per workspace, not per key**, enforced at the DB layer by a partial unique index `workspace_membership_one_bot_per_workspace on workspace_membership (workspace_id) where is_bot`. This is the race guard: two concurrent first-time provisions cannot create two bots - the loser gets a unique violation. Idempotency must NOT be check-then-insert in app code alone; the index is the hard guarantee.
- **Provisioning is `backend/support_bot.py:provision_workspace_bot(workspace_id) -> bot_user_id`** (async). It runs **lazily** (only when a caller invokes it - US-072 will call it on first widget-key issuance; nothing provisions a bot at workspace-creation time) and **idempotently** (a second call returns the existing bot's id, creates no second row). It creates the bot `auth.users` row via the GoTrue admin API (`POST {SUPABASE_URL}/auth/v1/admin/users`, requires `SUPABASE_SERVICE_ROLE_KEY` - resolved fail-closed at call time) then inserts the membership row; on a lost race it drops the orphan `auth.users` row and returns the winner. The returned id populates `conversations.bot_user_id`. The service-role key bypasses RLS - it is server-side only and never logged/returned/sent client-side.
- **Member-management listings MUST exclude the bot** (`where not is_bot`). It is an internal principal, not a human teammate, so the US-008 `/api/workspaces/{id}/members…` surface filters it out. This is a management/presentation contract, not a security boundary. (US-008's member endpoints are not present in this worktree yet; when they land they must add this filter - the migration documents it and the test pins it.) The schema also leaves room for an optional explicit write-deny policy on the bot row (e.g. a RESTRICTIVE `not is_bot` policy); not shipped, since the row is created/owned solely by the service-role path.
- **Optional env `SUPPORT_BOT_EMAIL_DOMAIN`** (default `bots.support.internal`) scopes the bot's internal, non-routable email; the row is admin-created with `email_confirm=true` and no password, so the address never logs in or receives mail.

Test: `python -m backend.test_us069_bot_provisioning` - a unit layer (always runs: workspace_id validation, fail-closed on missing `SUPABASE_URL`/`SUPABASE_SERVICE_ROLE_KEY` before any I/O) plus an integration layer (skips cleanly without local Supabase / the `is_bot` column / a reachable API): provisions twice and asserts exactly ONE `is_bot` row with `role='member'` and a stable returned id (one bot per workspace), no bot before provisioning, member-listing exclusion (`not is_bot`) with the human member as positive control, no `bot` role in the role CHECK, and the partial-unique-index race guard rejecting a second bot insert.

## Support-bot retrieval: minted-JWT principal, and a NON-security workspace filter (US-070, ADR-0008)

The support bot is **not** a privileged retriever. Each customer turn,
`backend/support_bot.py:run_bot_deflection_turn` mints a ~60s `role=authenticated`
JWT with `sub = bot_user_id` (US-068 `mint_supabase_jwt`, dependency-injected to avoid a
`main.py` import cycle), runs the ADR-0003 deflection pipeline with that JWT in the Supabase
headers, then discards the token (no cross-turn cache). So `match_chunks`/`keyword_search` resolve
`auth.uid()` to the bot and the **existing** membership + owner-OR-ACL boundary applies wholesale —
the bot sees only documents shared to it via `chunk_acl` (share-to-bot). There is **no new content
role and no new retrieval predicate** (ADR-0008: a new issuer beside GoTrue, not a new enforcement
path). The minted token is a bearer credential: it must never reach an SSE event, response body, or
log line — only `DeflectionResult.customer_message` is client-safe.

Sharp edge — **`filter_workspace_id` is NOT the trust boundary.** Migrations
`20260624150000` / `20260624150100` add `filter_workspace_id uuid default null` to
`match_chunks` / `keyword_search` as an *ordinary non-security narrowing filter*, AND-ed beside
`filter_topics` (CONTEXT "Active workspace"; reserved by the US-007 note). It can only **subtract**
within what the `auth.uid()`-resolved membership clause already allows; `null` is a no-op, so
`/api/chat` (which passes no `workspace_id`) and E4/E6 are byte-identical. The boundary is the
membership EXISTS clause, never this param. Both hybrid legs carry it so the narrowing is coherent.
Pinned live by `backend/test_us070_bot_retrieval_integration.py` (bot sees only the shared doc; the
filter narrows without leaking) and in isolation by `backend/test_us070_bot_retrieval.py` (mint
per-turn, bot bearer + filter on both legs, token never in the result).

## Opaque per-conversation customer token (US-071, ADR-0008 amends ADR-0004)

The anonymous customer's reconnect credential is an **opaque random token, NOT a Supabase JWT** - do not confuse it with US-068's `mint_supabase_jwt` (that mints the *bot's* identity token with `SUPABASE_JWT_SECRET`; this token is signed with nothing).
`backend/conversation_tokens.py` holds the pure primitives: `generate_conversation_token()` returns 256-bit `secrets.token_urlsafe` bytes; `hash_conversation_token(raw)` returns the SHA-256 hex that is the **only** representation stored.
The raw token is returned in the clear **exactly once** at conversation creation (only to the iframe, via US-078's first-message flow) and must NEVER hit a log line, SSE event, or any other response body.
Continuity comes solely from the iframe-origin-stored raw token - there is **NO** server-side customer-identity table.

This is a **THIRD trust boundary** beside the two above, and it is narrower: backend-mediated only.
`20260624160000_conversation_tokens.sql` adds `conversation_tokens(token_hash pk, conversation_id fk, expires_at, created_at)` with **RLS enabled and ZERO policies** (anon/authenticated denied wholesale), plus a `resume_conversation(p_token_hash text, p_slide boolean default true)` SECURITY DEFINER RPC granted to `service_role` **only**.
The customer presents the raw token to the backend over HTTP (the `X-Conversation-Token` header, deliberately not `Authorization`); the backend hashes it and calls the RPC as the service role.
The RPC is the single authoritative read path: it resolves the hash to its bound conversation iff `expires_at > now() AND status <> 'resolved'`, and returns the **one** conversation - no caller-supplied id reaches it, so a token for X structurally cannot read Y.
`p_slide` gates only the activity refresh: `true` (default) slides the 24h window (the POST /resume path); `false` resolves+gates identically but skips the `expires_at` UPDATE, so the read-only GET transcript path never mutates token lifetime.
`workspace_membership.role` appears nowhere here (ADR-0002 intact).

- **Supabase default-privileges gotcha:** `revoke execute ... from public` is NOT enough - Supabase grants EXECUTE on new `public` functions to `anon`/`authenticated` **directly** (not via PUBLIC), so the migration revokes from `public, anon, authenticated` by name, then grants to `service_role`. Verify with `has_function_privilege('anon', 'public.resume_conversation(text, boolean)', 'EXECUTE')` = false.
- **Invalidated on resolve, twice over:** the RPC's `status <> 'resolved'` gate plus an AFTER-UPDATE trigger (`conversations_purge_tokens_on_resolve`) that **deletes** the conversation's token rows when it transitions into `resolved` (literal invalidation + bounded growth). The BEFORE-UPDATE US-067 status guard has already validated the transition by the time it fires.
- **Lifetime ~24h**, slid forward on every successful *POST /resume* while not resolved; refresh extends `expires_at` on the *same* token (the raw token is stable across reloads - it is "returned once", never rotated). A GET transcript read is **not** activity and never slides the window.
- Endpoints (`main.py`): `POST /widget/conversations/resume` and `GET /widget/conversations/{id}/transcript`, both authed by the opaque token via `_resume_conversation_by_token`. POST resumes with `slide=True` (activity refresh); the **GET transcript path passes `slide=False`** so a nominally-safe/idempotent GET (browser prefetch, link-preview crawler, transparent retry) can never extend the token's lifetime as a side effect of the binding check. The transcript endpoint enforces the binding (resumed conversation id must equal the path id) so "token-for-X requesting Y" is rejected, and constrains the message read to customer-visible roles (`role=in.(user,assistant)`). Both helpers fail closed via `_require_service_role_headers()` (503 when `SUPABASE_SERVICE_ROLE_KEY` is unset). `_issue_conversation_token(http, conversation_id)` is the issuance helper US-078 will call on first message. Public-widget CORS is US-074's concern; these routes do not widen the `/api/*` `FRONTEND_ORIGINS` posture.

Test: `python -m backend.test_us071_conversation_tokens` - a unit layer (always runs: token is opaque/unique/URL-safe/not-JWT-shaped, hash is one-way 64-hex `!=` raw, 24h TTL) plus an integration/security layer (skips cleanly without local Supabase) encoding the PRD validation test: resume(Tx) returns X only (resume(Ty)->Y positive control), Tx can never resolve to Y, anon cannot call the RPC (service-role can), expiry/refresh behave, and resolving X purges Tx and rejects its resume.

## Widget keys: the non-secret public key + the not-revoked resolution gate (US-072, ADR-0008)

`widget_keys` (`supabase/migrations/20260624130000_widget_keys.sql`) is the public widget's key registry.
`public_key` is **NON-SECRET** - it is embedded verbatim in the buyer's page JS (the loader `<script>`, US-083), stored in the clear (nothing to hash, unlike US-071's `conversation_tokens.token_hash`), and grants NO access by itself: it only NAMES which workspace's bot a widget speaks to.
`backend/widget_keys.py` holds the pure primitives (`generate_public_key()` -> `wk_pk_<urlsafe>`; `is_widget_public_key()` shape guard); the leaked-key blast radius is the already-public KB, and the hard abuse controls are the rate limit + circuit breaker (US-076/077), NOT the key's secrecy.

This is a **FOURTH** boundary beside the three in this file, and it is the **admin-management** surface:

- **`role='admin'` legitimately enters the RLS here - the ONE place it may.** ADR-0002 bars `role` from *retrieval/visibility* predicates, but issuing/rotating/revoking keys IS the administrative action the admin role exists for. So the `widget_keys` SELECT/INSERT/UPDATE policies gate on the membership clause WITH `and wm.role = 'admin'`. There is intentionally **NO DELETE policy** (rotate = issue-new + revoke-old; a revoked key is retained for audit, deny-by-default delete).
- **Two faces on different CORS/auth surfaces.** Admin management (`/api/support/widget-keys*`, US-090's caller) runs INSERT/SELECT/UPDATE under the admin's *own* JWT so the admin RLS IS the authorization (a non-admin's write is rejected by Postgres, not app code). Public resolution (`POST /widget/keys/resolve`) is anonymous and reads under the **service role** (the widget holds no Postgres role), gating on `revoked_at IS NULL` and leaking no workspace topology. Public-widget CORS is US-074, per-key origin enforcement US-073, rate-limiting US-076 - all layer on top without changing this gate; these routes do not widen the `/api/*` `FRONTEND_ORIGINS` posture.
- **Resolution is the not-revoked gate.** `_resolve_widget_key` is a service-role read filtered by `public_key=eq AND revoked_at=is.null`; a revoked/unknown key matches **zero rows** and mints nothing. Revoking blocks NEW conversations but **never terminates a live one** - the opaque per-conversation token (US-071) is independent of the key once minted.
- **Revocation is a one-way DB-enforced latch**, the same "enforced in the DB, not just the service layer" stance as the US-067 status machine. A `before update` trigger (`public._widget_keys_revoke_guard` / `widget_keys_revoke_guard`) rejects any mutation of an already-set `revoked_at` (cannot clear to NULL, cannot move the timestamp), so a revoked key can never be re-activated by a direct PostgREST PATCH behind the admin UPDATE policy (which is column-unrestricted on purpose). `NULL -> timestamp` (the revoke action) and `NULL -> NULL` (label/origin edits) stay allowed; admins may still freely edit `label`/`allowed_origins` even on a revoked row.
- **First-key-issuance bot provisioning.** `_ensure_workspace_bot` (main.py) is the single, idempotent, best-effort call site for US-069's `support_bot.provision_workspace_bot(workspace_id, http=...)` (`workspace_id` is the only positional; `http`/url/key are keyword-only). First issuance provisions the per-workspace bot; later issuances return the same id. Provisioning is best-effort - a failure (or a build without the support-bot module) is logged and never blocks key issuance, and US-069 is also triggered lazily at first conversation (US-078).

Test: `python -m backend.test_us072_widget_keys` - a unit layer (always runs: public key prefixed/unique/URL-safe, shape guard rejects blank/prefix/garbage) plus an integration/security layer (skips cleanly without local Supabase) encoding the PRD validation test: active K resolves to its workspace while revoked Kr resolves to 0 rows (no minting), an admin reads both keys + can insert/revoke while a non-admin member and a non-member read 0 and cannot write, and revoking Kr leaves the live conversation + its US-071 token fully resumable.

### Per-key registered-origin allowlist, fail-closed (US-073, ADR-0008)

`widget_keys.allowed_origins text[]` (already added by US-072's migration - **no new migration**) carries a per-key registered-origin allowlist enforced at public resolution.
`backend/widget_keys.py:is_origin_allowed(origin, allowed_origins)` is the pure, always-unit-testable matcher; `main.py:widget_resolve_key` reads the request `Origin` header (added a `request: Request` param) and refuses with the SAME opaque 404 (`"unknown or inactive widget key"`) when it returns False, so nothing leaks whether the key exists or which origins it allows. The not-revoked gate (`_resolve_widget_key`) stays FIRST; origin is an additional gate layered on top, ordered after, so a revoked key and an unlisted origin are indistinguishable in the response.

- **Fail-closed by construction, never fail-open** - every ambiguous case refuses: an empty `[]` OR null allowlist makes the key **INACTIVE** (a key with no registered origin resolves nothing); a missing OR blank `Origin` is refused (the cross-origin widget always sends one - absence means refuse, even under the wildcard). `[]`/null (empty, inactive) is kept distinct from `["*"]` (non-empty, active wildcard).
- **Issuance refuses a dead-on-arrival key (issue #36, US-073 follow-up).** `widget_keys.py:has_registered_origin(allowed_origins)` is the pure WRITE-side mirror of that empty-allowlist resolution check: it returns True iff at least one entry is non-blank (the dev-only `"*"` counts as a usable, deliberately permissive entry). `main.py:issue_widget_key` (`POST /api/support/widget-keys`) calls it FIRST and rejects an empty/null/blank-only `allowed_origins` with a hard **400** BEFORE generating a key or provisioning the bot, so an admin can never mint a key US-073 would render silently INACTIVE. Like the resolution gate this is defense-in-depth UX, **NOT a security boundary** - the gate already stops an originless key from working; the guard just makes the failure loud at creation instead of a surprise later. It asserts only that the key is *potentially-functional*, not that any origin is well-formed - comparison stays exact/un-normalized, so a mis-typed origin still fails CLOSED at resolution.
- **`"*"` is a documented DEV-ONLY opt-in (PRD F3 row), non-production**: a key whose `allowed_origins` contains `"*"` matches any *present* origin (`WILDCARD_ORIGIN` in `widget_keys.py`). It still refuses an originless request.
- **Comparison is exact, un-normalized string membership** - origins are scheme+host[+port], no path/trailing slash, exactly the form the browser emits in `Origin` (which it already lower-cases). Any mismatch (casing, stray slash, port) fails CLOSED - the safe direction - so admins must register origins in that canonical form.
- **Defense-in-depth, NOT a hard control.** The `public_key` is non-secret and `Origin` is trivially forgeable off-browser, so this only blunts casual key-lifting and in-browser cross-site abuse. The hard abuse controls remain the rate limit + circuit breaker (US-076/077); the leaked-key blast radius is the already-public KB. Do not overstate it as a security boundary. Scope is the `/widget/keys/resolve` path; US-078 (lazy conversation creation) re-resolves server-side and **reuses the same `is_origin_allowed` helper**.

### Two CORS postures on one app, partitioned by path (US-074, ADR-0008)

The public widget surface has its OWN CORS posture, **independent of the authenticated app's `FRONTEND_ORIGINS`**.
The single global `CORSMiddleware(allow_origins=FRONTEND_ORIGINS)` is replaced by **two complementary, path-scoped CORS layers** on one app (`main.py`):

- the **authenticated** posture (`_ScopedCORSMiddleware(owns_widget=False, allow_origins=FRONTEND_ORIGINS, …)`) owns everything that is NOT a public widget route - `/api/*`, `/healthz`, etc. - keeping EXACTLY the prior posture (same `FRONTEND_ORIGINS`, same methods, `allow_headers=["*"]`); and
- the **public-widget** posture (`_WidgetCORSMiddleware`, owns `/widget/*`) trusts ONLY origins registered on an ACTIVE (not-revoked) widget key - the union of every `widget_keys.allowed_origins` - and **never** `FRONTEND_ORIGINS`.

The two origin sets are independent: an app origin is rejected at `/widget/*`, and a widget origin is rejected at `/api/*` (the PRD validation test). The partition is the single predicate `_is_widget_public_path(path)` so the two layers are guaranteed complementary - every request is owned by exactly one posture (no double `Access-Control-Allow-Origin`, no gap).

- **CORS is a BROWSER-side control and DEFENSE-IN-DEPTH here, NOT the hard boundary.** A preflight (`OPTIONS`) carries no body, so the specific `public_key` is unknown at preflight time - this layer can only answer "is this Origin registered for SOME active key". The authoritative PER-KEY origin gate (US-073) + not-revoked gate (US-072) + rate limit (US-076) run inside the endpoints under the real `public_key`. Letting the browser READ a `/widget/*` response from a registered origin grants no data access by itself.
- **Subclass, don't reimplement.** `_ScopedCORSMiddleware(CORSMiddleware)` passes through requests it does not own (so two instances run independent postures with no overlap); `_WidgetCORSMiddleware` overrides `is_allowed_origin` to consult the dynamic snapshot instead of a static list, and constructs with `allow_origins=()` so the reflected `Access-Control-Allow-Origin` is ALWAYS the specific request Origin, **never the literal `*`** - even the dev-only `"*"` key opt-in only widens WHICH origins are admitted, never the reflected value (AC: "never `*` outside dev opt-in").
- **Dynamic origins via a TTL-cached snapshot, fail-closed.** `_WIDGET_ORIGIN_SNAPSHOT` (`_WidgetOriginSnapshot`) caches the union of active-key origins (+ a wildcard flag) for `WIDGET_CORS_ORIGIN_CACHE_TTL` seconds (default 30) so a preflight / simple request never pays a service-role round-trip per call (CORS runs AHEAD of the US-076 rate limit, so a per-request DB hit would be a cheap DoS lever). `_load_active_widget_origins` FAILS CLOSED - support unconfigured (no service-role key) or an upstream error returns `(empty, False)` so the surface DENIES every origin rather than fail-open (logged concisely, no per-tick traceback). `allows()` mirrors `is_origin_allowed`'s rules at the CORS layer: a missing/blank Origin is refused even under `"*"`; otherwise exact, un-normalized membership (trailing-slash/case/port mismatch fails closed). Issuing/revoking a key calls `_WIDGET_ORIGIN_SNAPSHOT.invalidate()` so a newly registered origin works (and a revoked one stops) on the NEXT preflight on this instance; the TTL is the cross-instance backstop. Invalidation is race-safe against an in-flight refresh via a generation guard: `invalidate()` bumps `_generation` (and clears `_loaded_at`), and `ensure_fresh()` captures the generation before its `await` and stamps the load fresh only if the generation is unchanged afterward - so an invalidate firing mid-load (the load having read pre-mutation DB state) forces a reload instead of caching the stale snapshot for the full TTL. (`WIDGET_CORS_ORIGIN_CACHE_TTL` must be `> 0`: 0 makes `_fresh()` always False and turns every `/widget/*` request into a serialized service-role DB read, the per-request-DB DoS lever the cache exists to prevent.)
- **Authenticated routes under `/widget/*` are excluded from the public partition.** `_WIDGET_AUTHENTICATED_SUFFIXES` (currently `("/agent-reply",)`) carries `/widget/*` routes that are authed by a real Supabase JWT and called from the operator dashboard (an APP origin), NOT the anonymous-customer surface. US-082's `POST /widget/conversations/{id}/agent-reply` is the known case; it falls to the authenticated posture (the dashboard's app origin would otherwise be CORS-blocked by the per-key allowlist). US-074's public surface is exactly the AC-enumerated anonymous routes (key resolution, conversation create/message, transcript, customer SSE).

## Swappable `RateLimiter` seam: durable Postgres default, no in-memory backend (US-075, ADR-0008)

`backend/rate_limiting.py` is the abuse-counter seam the public widget's hard controls draw down against (US-076 per-key + per-session/IP windows, US-077 per-workspace circuit breaker). It mirrors the repo's other provider factories (`reranking.build_reranker`, `web_search.build_web_search_provider`, `parsing.build_parser`) - a `RateLimiter` ABC + an env-selected factory - so a call site is byte-identical regardless of backend. **This story ships ONLY the seam + migration; it has no call site yet by design** (US-076 wires it onto the endpoints).

- **The seam is the ABC; the backend is a `RATE_LIMITER` flip.** `RateLimiter` exposes `hit(key, *, limit, window_seconds, cost=1) -> RateLimitDecision{allowed,count,limit,window_seconds}`, a read-only `count(key, *, window_seconds)` peek, and `aclose()`. `get_rate_limiter_name()` reads `RATE_LIMITER=postgres|redis` (default `postgres`); `build_rate_limiter()` is the factory. "Swap is call-site-free" is structural: every caller talks only to the ABC, so the `name` flip is the only change. The factory **fails closed at build time** (like the reranker/web-search factories raising on missing API keys): postgres needs `http` + `SUPABASE_URL` + `SUPABASE_SERVICE_ROLE_KEY`; redis needs `REDIS_URL` + the optional `redis` package (lazily imported, clear actionable `RuntimeError` when absent).
- **There is intentionally NO in-memory / `none` backend, and that is load-bearing (ADR-0008).** An in-process counter under-counts per replica (each instance counts only its own slice, so N replicas admit up to N× the rate) and resets on restart (a deploy becomes a free abuse window) - both defeat the entire point of a cost-DoS guard. So `RATE_LIMITER` rejects anything but `postgres|redis`; durable + cross-instance is the floor, not an upgrade. Recorded in the module docstring on purpose.
- **Postgres default = service-role RPCs over PostgREST, same posture as US-071's `resume_conversation`.** `PostgresRateLimiter` never opens a Postgres connection; it calls two SECURITY DEFINER RPCs over PostgREST under the service role. Migration `supabase/migrations/20260624170000_rate_limit_counters.sql` adds `rate_limit_counters(bucket_key, window_seconds, window_start, count, pk(bucket_key, window_seconds, window_start))` - `window_seconds` is part of the row identity so the same key under two different window sizes never conflates and the window-relative prune stays scoped to one (key, window), making the Postgres and Redis backends isolate by (key, window) identically (Redis already namespaces via `{prefix}:{key}:{window_seconds}`) - with **RLS enabled and ZERO policies** (anon/authenticated denied wholesale), plus `rate_limit_hit(p_key, p_limit, p_window_seconds, p_cost default 1)` / `rate_limit_count(p_key, p_window_seconds)` - revoked from `public, anon, authenticated` **by name** (the Supabase default-privileges gotcha: EXECUTE is granted to anon/authenticated DIRECTLY, not via PUBLIC) then granted to `service_role` ONLY. Verified live: `has_function_privilege('anon', …)` = false and an anon PostgREST call returns `42501 permission denied`. `workspace_membership.role` appears nowhere (ADR-0002 intact) - a rate-limit key is an opaque string (`"key:<public_key>"`, `"ip:<addr>"`, `"ws:<workspace_id>"`) the backend composes; this store assigns it no meaning and enforces no boundary, it only counts.
- **Algorithm = two-bucket sliding-window counter, bounded, write-cheap.** Both backends use the textbook sliding-window counter: the current fixed bucket's exact count + the previous bucket weighted by the fraction still inside the trailing window (smooths the fixed-window edge burst into a proper sliding bound). The Postgres weighting lives in the RPC (numeric epoch math, `to_timestamp(floor(epoch/window)*window)` bucket labels); an opportunistic in-RPC prune keeps it bounded to **≤2 live rows per (key, window)** - deliberately NOT a sliding-window *log* (one row per hit), which would amplify writes precisely under the abuse it exists to dampen. A blocked hit STILL increments (keeps a hammering caller saturated; a backed-off caller recovers as the window slides), and the decision + count return together so a caller never re-reads (no TOCTOU). The Redis adapter (optional, `redis` NOT in `requirements.txt`) mirrors the same weighting on its OWN clock - a documented cross-instance-skew approximation, the caveat any distributed limiter carries - with `2*window` key TTLs and a Lua incr+read for atomicity.

### Two-window rate limit wired onto the widget surface (US-076, ADR-0008)

US-076 is the first **call site** of the US-075 seam: `main.py` draws every public-widget request down against TWO sliding windows so the surface (which drives PAID retrieval + LLM draft/judge calls) cannot be turned into a cost-amplification DoS. The seam stays untouched - this story only wires it on; the per-workspace circuit breaker is still US-077.

- **Two windows via TWO single-window helpers, charged with a short-circuit between them (NOT one `asyncio.gather`).** `_charge_widget_window(bucket_key, *, limit, scope, cost)` is the shared core; `_enforce_widget_session_limit(request, *, cost=1)` charges the **per-session/IP** bucket `ip:<session>` (caps one caller hammering across keys) and `_enforce_widget_key_limit(public_key, *, cost=1)` charges the **per-key** bucket `key:<public_key>` (caps aggregate abuse of one key across EVERY session/IP - so a fresh session under a hammered key is still refused), both at `WIDGET_RATE_LIMIT_WINDOW_SECONDS` (default 60s) with limits `WIDGET_RATE_LIMIT_PER_KEY` (300) / `WIDGET_RATE_LIMIT_PER_SESSION` (30). Each window is its own AND-ed gate: a request is refused if EITHER is over, and a blocked hit still counts (keeps a hammering caller saturated). The split (vs one concurrent `gather` over both) is what lets the endpoint short-circuit: the per-SESSION window is charged FIRST, before the DB resolve; the per-KEY window is charged only AFTER the resolve proves the key real. **This is the storage-amplification fix (US-076 review):** a non-existent/fake key never reaches the per-key charge, so a rotating-`wk_pk_<random>`-key attacker can never mint a permanent `rate_limit_counters` row (the per-key bucket is created only for the small, controlled set of keys that actually exist); a session-throttled request mints no per-key row and does no resolve. The residual unbounded axis is `ip:<spoofed-XFF>` session rows - exactly the off-browser traffic the P5 edge/WAF complement handles - and a global TTL/sweeper on `rate_limit_counters` is intentionally left to the US-075 seam (out of US-076 scope). Both windows must enforce independently: the validation test pins that an exhausted session is refused while the key still has headroom, AND a fresh session is refused by an exhausted per-key window (the "only one window enforces" failure indicator).
- **Fail OPEN on a limiter-backend error (the counter store is NOT a request-path SPOF).** `rate_limit_counters` live in the SAME Postgres as the resolve/retrieval path, so `_charge_widget_window` wraps `limiter.hit` in a try/except: ANY backend error logs a concise `widget_rate_limit.limiter_error scope=… — failing open` warning (no per-request traceback flood) and ALLOWS the request rather than letting a transient counter-RPC glitch 500 the entire public widget surface. Availability wins; the edge/WAF limiter (P5) is the hard bound. Only backend/limiter errors fail open - a genuine over-limit `HTTPException(429)` is raised OUTSIDE the try, so it is never swallowed.
- **429 throttle, DISTINCT from the US-077 circuit breaker.** A breach raises a hard **429** with a `Retry-After` header having done NO costly work - the per-session gate is invoked AFTER the cheap `is_widget_public_key` shape guard (a malformed key is a free 404, not worth a counter write) but BEFORE the DB resolve / retrieval / LLM, and the per-key gate runs after the cheap resolve existence-check but still before any retrieval/LLM / the `{active}` reply. This is a "slow down, retry the same request" refusal; US-077's tripped breaker is a **200** generic "a human will follow up" deferral (no retry needed). The response body is generic (does not echo which window tripped); the scope (`key`/`session`) is logged for ops, not returned, so nothing leaks per-key/per-IP topology.
- **Session identity is best-effort + spoofable on purpose.** `_widget_client_ip` prefers the LEFT-most `X-Forwarded-For` hop (the original client, since prod sits behind a proxy and `request.client.host` is the proxy), falling back to the socket peer then `"unknown"`. `X-Forwarded-For` is client-spoofable, so the per-session/IP window is **defense-in-depth**: a header-rotating attacker sidesteps it, but the per-key window still aggregates their abuse of a real key, and the **edge/WAF limiter is the documented production complement (P5)** that stops spoofed traffic before the app. App-level limiting is the portable default; v1 accounting is COARSE (requests-per-window, `cost`-weighted - NOT precise token/$ metering, the F3 future refinement). `cost` lets a heavier endpoint (US-078's message turn) charge more than a cheap key-resolution open.
- **Lifecycle + no-op-when-unconfigured.** The limiter is built ONCE at startup via the US-075 factory (`get_rate_limiter_name()` + `build_rate_limiter()`) only when `SUPABASE_SERVICE_ROLE_KEY` is set, and fails CLOSED at build time on a misconfigured backend (loud boot failure beats a silently-unprotected path). When support is unconfigured the limiter is `None` and enforcement is a clean **no-op** - safe because every widget endpoint already 503s (via `_require_service_role_headers`) before reaching anything costly, so a built limiter IS the enforcement and its absence == an inert surface, never an unprotected one. The Postgres backend borrows a long-lived `_RATE_LIMITER_HTTP` client this module owns; `_on_shutdown` closes the limiter (Redis closes its own client; Postgres `aclose` is a no-op) then that client. Wired onto `POST /widget/keys/resolve` now (per-session first, per-key after the resolve); US-078's first-message conversation-create path re-runs the same two helpers (with a higher `cost`) on the message surface.

### Per-workspace circuit breaker → zero-cost deferral + operator badge (US-077, ADR-0008)

`backend/circuit_breaker.py` is the per-WORKSPACE cost/qps breaker, **orthogonal to US-076's per-key/per-session windows**: US-076 caps *who* may hammer the surface; the breaker caps one workspace's whole-bot aggregate budget, so even legitimate-looking distributed traffic (many sessions, many real keys) cannot run a workspace's LLM bill away. It draws down the **SAME US-075 `RateLimiter` seam** US-076 uses, on a distinct `ws:<workspace_id>` bucket - **no new store, no new migration** (it reuses `rate_limit_counters`); the seam still only counts, the per-workspace MEANING lives in this module. The bucket namespace is deliberately distinct from US-076's `key:`/`ip:` so the ceilings never conflate even at the same window size.

- **Zero-cost when tripped is STRUCTURAL, not a flag.** `run_breaker_guarded_turn(*, limiter, workspace_id, limit, window_seconds, run_turn, on_trip=None, cost=1) -> GuardedTurnResult` checks the breaker FIRST and, when tripped, **NEVER awaits the injected `run_turn` thunk** - so the (paid) ADR-0003 deflection pipeline makes **zero retrieval and zero LLM calls**. The short-circuit is the turn simply not being invoked, never a flag the pipeline checks partway through. `check_workspace_breaker(limiter, workspace_id, *, limit, window_seconds, cost=1) -> BreakerDecision{tripped,count,limit,window_seconds}` is the lower primitive (`tripped = not allowed`; a blocked hit still counts).
- **FAIL-OPEN, even more clearly than US-076.** NO-OP (never trips) when the limiter is `None` (support unconfigured - the widget endpoints 503 before anything costly), and FAIL-OPEN (never trips) on ANY limiter-backend error. The asymmetry vs US-076: a per-request 429 failing open just skips one throttle, whereas a **breaker failing CLOSED would defer a workspace's ENTIRE traffic** (a self-inflicted DoS), so a transient counter-RPC glitch must never convert every turn into a deferral. Availability wins; the edge/WAF limiter (P5) is the hard bound.
- **Escalate, don't "resolve" - so a trip is NEVER counted as a deflection.** A tripped breaker is a human HANDOFF: `run_breaker_guarded_turn` returns `GENERIC_BREAKER_DEFERRAL` with `turn=None` (**no partial/garbage answer**) and fires the optional `on_trip` hook, which US-080 wires to the US-067 status latch (a plain Postgres write to `escalated`/`escalated_at`). Because `escalated_at` is then set, the derivable deflection metric (`resolved AND escalated_at IS NULL` => deflected) reads the conversation as **human-handled**, never a deflection. `on_trip` is dependency-injected (like `support_bot.run_bot_deflection_turn`'s minter) so this control-flow seam stays pure/unit-testable with no DB/LLM and US-080 supplies the concrete escalation write; its exceptions are NOT swallowed (escalation is significant; the runtime owns error handling).
- **Operator badge = in-app Realtime, ZERO outbound (same posture as escalation-notify).** "Emit a badge" is NOT an outbound call: the trip escalates the conversation in Postgres and the operator dashboard (US-087 `/support/queue`) watches `status='escalated'` live via the agent's OWN Supabase Realtime `postgres_changes` under the agent's real JWT. No ESP, no webhook, no Slack anywhere in the path - the "zero outbound" is structural (the trip path touches no network client).
- **The customer-facing deferral is generic ON PURPOSE.** `GENERIC_BREAKER_DEFERRAL` promises a human and leaks NOTHING operational - no workspace id, no counts, no "rate limited"/"breaker"/"ceiling"/429. It differs from escalation's retrieval/faithfulness deferral only in user-facing framing ("high volume" per the PRD wording); an attacker can confirm the workspace breaker tripped no more than the US-076 429s already reveal rate-limiting exists. (A stricter alternative - reuse escalation's exact `GENERIC_DEFERRAL` so a customer cannot even distinguish a breaker trip from a normal escalation - was considered; the "high volume" framing was kept for honesty/UX since it leaks nothing more.)
- **Wiring + the no-live-call-site-yet stance.** `main.py` adds env knobs `WIDGET_BREAKER_PER_WORKSPACE` (default 600, deliberately ABOVE organic single-workspace traffic - a cost-runaway backstop, NOT the everyday throttle, which is US-076) and `WIDGET_BREAKER_WINDOW_SECONDS` (defaults to the US-076 window), feeding `_check_workspace_breaker(workspace_id, *, cost=1) -> BreakerDecision` which delegates to the module against the SAME `_RATE_LIMITER` built once at startup. Unlike the US-076 `_enforce_widget_*` helpers (which RAISE a 429), this RETURNS a decision: a trip is a 200 generic deferral + handoff, not a "retry the same request" refusal, so the caller short-circuits and escalates rather than rejects. **Like the US-075 seam, this story ships the breaker + guarded runner + `_check_workspace_breaker` with NO live call site yet by design** - the message-turn runtime (US-078 lazy conversation create / US-079 deflection streaming / US-080 escalation latch) invokes `run_breaker_guarded_turn` on the message path with a real turn thunk + the escalation write as `on_trip`.

### Deterministic deflection turn streams over the request-scoped SSE (US-079, ADR-0003/0008)

`POST /widget/conversations/messages` (US-078 opened the request-scoped SSE; US-079 makes the bot actually answer over it). The botless short-circuit happens BEFORE any import/mint/LLM (it just returns), so a knowledge-assistant-only or half-provisioned deployment degrades to "a human will follow up" rather than 500-ing the widget.

A tripped breaker NEVER awaits the turn thunk (zero retrieval, zero LLM - the structural short-circuit) and streams `GENERIC_BREAKER_DEFERRAL`. **SCOPE BOUNDARY:** US-079 passes `on_trip=None` and writes NO status; the escalation LATCH (`status='escalated'` / `escalated_at`), the breaker's `on_trip` escalation write, and the "skip the pipeline once escalated" routing are ALL US-080. So a US-079 breaker trip is cost-safe (the primary job) but not yet a recorded handoff - that gap closes in US-080. The conversation stays `active` on every US-079 branch. Complementing the breaker's workspace-wide ceiling, the SSE generator RE-CHECKS `request.is_disconnected()` immediately before the paid turn: a client that dropped right after the `conversation` event incurs ZERO retrieval/LLM and persists no reply, trimming the obvious per-request waste the aggregate ceiling alone would not catch.
- **Streaming is REPLAY-chunking, not live token streaming - and it must be.** The ADR-0003 pipeline gates the WHOLE drafted answer on the faithfulness judge (US-048) BEFORE it may send, so the draft cannot be streamed token-by-token live (a half-streamed answer the gate later rejects would already have leaked). The answer is computed in full, then `_split_for_streaming(text)` replays it as a handful of word-boundary `delta` events for incremental render. `"".join(_split_for_streaming(t)) == t` for any `t` (whitespace separators preserved), so the streamed text is byte-exact. `conversation_messages.tool_calls` stays null (the deterministic pipeline is control flow, NOT the agentic tool loop - AC4; the `_conversation_message_insert_payload` builder omits the field).
- **The bot reply is persisted BEFORE the deltas stream** (same durability rationale as US-078's pre-SSE user-message persist): the transcript reflects the reply even if the client disconnects mid-stream, and what streams is exactly what is recorded. A persistence failure surfaces as an `error` event (no deltas), so the transcript and the stream never disagree.
- **Resume needs `bot_user_id`, which the US-071 `resume_conversation` RPC does not return** (its customer-facing view is id/workspace_id/status/created_at only). Rather than DROP/recreate that security-DEFINER RPC (and re-apply its grants) to widen the return, US-079 reads `bot_user_id` under the service role via `_load_conversation_bot_user_id(http, conversation_id)`, keyed by the id the opaque token already resolved to - so a RESUMED conversation still gets bot answers. The first-message branch already has `bot_user_id` in hand (the INSERT representation), so this read is the resume path only. `_load_conversation_bot_user_id` is itself FAIL-CLOSED: it never raises, returning None (logged) on a transient read error as well as for a genuinely botless workspace, so a recoverable read glitch degrades to the generic deferral rather than 500-ing the customer surface (mirroring the first-message path's `_ensure_workspace_bot`, which never raises).
- **Rate limiting is unchanged from US-078** (US-076 two-window charge at `WIDGET_RATE_LIMIT_MESSAGE_COST`); US-079 adds no new charge. The per-workspace breaker draws down a distinct `ws:` bucket (US-077), orthogonal to the `key:`/`ip:` windows.

### Escalation latch routes all later customer messages to the human (US-080, ADR-0004/0008)

US-080 makes the *first escalate decision* a one-way latch: once a conversation flips to `status='escalated'` the bot goes silent FOREVER for that conversation, so a human who has been engaged can never have the bot auto-send a confident (possibly wrong) answer into the thread behind their back (PRD Risk #3). It closes the SCOPE BOUNDARY US-079 left open - US-079 made a breaker trip *cost-safe* but not yet a *recorded handoff*; US-080 records the handoff and routes around the pipeline thereafter. **No new migration, no new env** - it reuses the US-066 `conversations` table and the US-067 status machine.

- **`_escalate_conversation(http, conversation_id) -> dict | None` is the SINGLE latch path** for BOTH the model-mediated escalate AND the explicit "talk to a human" button (US-091) - escalate-vs-answer is deterministic control flow, NEVER a model `escalate()` tool (ADR-0003). It is a service-role PATCH that sets **ONLY** `status='escalated'`; **`escalated_at` is NEVER planted by the caller** - the US-067 BEFORE-UPDATE trigger owns it (stamps `now()` on the FIRST transition into escalated, preserves it verbatim afterward), which is exactly what keeps the derivable deflection metric correct (`resolved AND escalated_at IS NULL` => deflected). Idempotent: `escalated -> escalated` is a DB no-op the US-067 guard allows. Backend-mediated under the service role (the anonymous customer holds no Postgres role; the ADR-0004 membership RLS is bypassed as for US-078 conversation creation), keyed by an id the caller already resolved - never caller-supplied.
- **Only a DELIBERATE escalate latches; a recoverable degraded deferral does NOT.** Inside `_run_widget_bot_turn` two deliberate sources latch (best-effort, via `_escalate_conversation_safe`): a **tripped US-077 breaker** - wired as the breaker's `on_trip` hook, the hook US-077 shipped for exactly this (US-079 passed `on_trip=None`) - and the **pipeline returning `action='escalated'`** (weak retrieval / unfaithful draft), latched after the un-tripped turn. The fail-closed DEGRADED deferrals (botless workspace, unimportable `support_bot`, any pipeline exception) deliberately **do NOT latch**: they may be transient, and permanently silencing the bot on a blip would be wrong - they defer THIS turn and leave the conversation `active` to recover next turn. A confident answer never latches. This preserves US-079's "botless escalates without touching the pipeline / http" invariant (the botless path still returns before any write).
- **The in-turn latch is BEST-EFFORT; the explicit endpoint is STRICT.** `_escalate_conversation_safe` swallows a latch-write failure with a concise warning (no per-request traceback flood - same convention as the rate-limiter/breaker fail-open paths) so a transient write glitch never turns the deferral 200 into a 500; the conversation stays `active` and re-evaluates next turn. The user-pressed `POST /widget/conversations/escalate` instead calls the strict `_escalate_conversation` and surfaces a failure as **502** so the button can prompt a retry.
- **Routing: an already-escalated conversation NEVER re-runs the pipeline (AC2).** `widget_conversation_message` captures `conversation_status` (a created row is always `active`; a resumed row carries its real status) and the SSE generator short-circuits a `status=='escalated'` conversation to `conversation` + `done` - the customer message was already persisted before the SSE opened (the queue input), the bot stays silent, and the human reply arrives asynchronously over the US-081 SSE. The resume branch also SKIPS the `_load_conversation_bot_user_id` service-role read once escalated (the bot principal is never needed). The FIRST escalating turn still runs the pipeline (to get the escalate decision + emit the deferral) and latches as a side effect; only LATER messages skip.
- **In-flight race guard: a concurrent escalation mid-pipeline still silences the bot (Risk #3, review follow-up).** The entry-snapshot short-circuit catches the SEQUENTIAL later-message case but not the CONCURRENT one: the deflection pipeline is seconds of LLM work (retrieval + draft + faithfulness judge), and a customer pressing "talk to a human" (`POST /widget/conversations/escalate`) or another message can latch `status='escalated'` while THIS turn is mid-flight. So after `_run_widget_bot_turn` returns but BEFORE persisting/streaming the reply, the generator RE-READS the live status via fail-open `_load_conversation_status(http, conversation_id)` and, if now `escalated`, SUPPRESSES a CONFIDENT answer entirely (no `assistant` row, no `delta` events, just `done` - the human owns the thread, and the already-persisted user message is their queue input). The guard fires ONLY for a confident answer: the deferral replies (`GENERIC_DEFERRAL` / `GENERIC_BREAKER_DEFERRAL`) are EXEMPT because they themselves promise a human and may BE this turn's own latch (the model-mediated escalate / breaker trip), which AC1 requires us to stream. Fail-open by design - a None status (transient read glitch, or `SUPABASE_SERVICE_ROLE_KEY` unset) streams normally rather than silencing a valid faithful answer, since the NEXT turn is latched anyway; it adds exactly one service-role read per confident-answer turn.
- **`POST /widget/conversations/escalate` is the explicit "talk to a human" latch path (AC3, US-091's button calls it).** Token-authed (US-071 opaque `X-Conversation-Token`, NOT a Supabase JWT - the anonymous customer is off the Supabase trust surface), per-session rate-limited (US-076; no `public_key` on this path, and the write is cheap, so per-session only), it resumes the token to its ONE bound conversation and latches via `_escalate_conversation`. A missing/expired/resolved token -> 401 (start fresh). It runs **no pipeline** - it is UI-initiated, deterministic, never a model tool. Under `/widget/*` and NOT a `_WIDGET_AUTHENTICATED_SUFFIXES` route, so it inherits US-074's public-widget CORS posture (the `X-Conversation-Token` header is already allowed there).

### Loader `<script>` + cross-origin iframe widget shell (US-083, ADR-0008)

The public widget is embedded as a **cross-origin iframe served from the kit's OWN origin** by a tiny vanilla loader (`frontend/public/widget.js`). That cross-origin boundary - NOT the loader - is the security control: the US-071 conversation token + session live in the iframe's own-origin `localStorage`, structurally unreadable by host JS or an XSS on it (a host read of `iframe.contentWindow.localStorage`/`document` throws `SecurityError`). Host↔widget communication is **`postMessage` only** - nothing is shared in-process - so theming and state can never bleed without crossing that boundary (ADR-0008 max-isolation embed). This is the embed **SHELL** only; the in-iframe chat UI + theming render is US-084. **Frontend-only: no backend, no migration, no env.**

- **The loader derives the kit origin from its OWN `<script src>`, never the host origin.** `new URL('widget.html', selfScript.src)` makes the iframe `src` the kit's `widget.html` regardless of where the buyer mounts the loader, so the iframe is always a different origin from the host page. It reads non-secret config from `data-*` attributes (`data-public-key` is the US-072 `wk_pk_…` key; plus `brand-color`/`greeting`/`title`/`position`/`launcher-icon`/`z-index`) overridable via `window.SupportWidgetSettings`, injects a fixed-corner iframe sized to a bubble (closed) / panel (open), and is idempotent (a double-include yields one widget). There is deliberately NO host-supplied `apiBase`: the widget talks only to its OWN build-time backend (the kit's origin), so a host-page XSS can never redirect where the US-071 token is sent (US-084 security fix; the iframe also accepts `init` exactly once, ignoring re-init). It exposes a **token-free** `window.SupportWidget` (`open`/`close`/`toggle`/`onUnread`) - the ONLY host global, and it carries no token by construction.
- **postMessage contract (`frontend/src/widget/protocol.ts` is the source of truth; the vanilla loader hand-duplicates the strings).** Every message carries `channel: "ar-support-widget@1"`; both sides ignore anything else. host→widget: `init` (public key + presentation config, sent in reply to `ready`, `targetOrigin` pinned to the kit origin) and `command` (`open`/`close`/`toggle`). widget→host: `ready` (the ONLY `targetOrigin: '*'` send - it carries no secret and the iframe does not yet know who embedded it), `state` (`{open}` - the iframe is the single source of truth; the loader resizes off this, so there is no open/close feedback loop) and `unread` (`{count}`, surfaced to the host via a `supportwidget:unread` CustomEvent + the optional `onUnread` callback for title/favicon use). Origin handling is strict BOTH ways: the loader acts only on `event.origin === kitOrigin && event.source === iframe.contentWindow`; the iframe pins the host origin from the first `init` and refuses any later message from a different origin.
- **EVERYTHING (launcher + panel) lives INSIDE the iframe, not the host DOM.** The loader only sizes/positions the iframe and relays messages, so host CSS cannot reach any pixel and the launcher itself cannot be restyled/spoofed by the host. A closed iframe is bubble-sized (96×96) so it never blocks clicks on the rest of the host page. The widget bundle is a SEPARATE Vite entry (`frontend/widget.html` → `src/widget/main.tsx`, added to `rollupOptions.input`) that never imports the admin app / supabase-js / dark theme; `dist/widget.html` + `public/widget.js` are real static files so the `vercel.json` SPA rewrite (fires only when no file matches) leaves them directly reachable.
- **Web-component / shadow-DOM embed DELIBERATELY REJECTED (AC4).** Shadow DOM encapsulates markup/styles but runs in the host's SAME JavaScript realm, so host JS could read the token off the element's state/storage - cosmetic encapsulation, never a trust boundary. The cross-origin iframe is the embed instead. Noted in `widget.js`, `protocol.ts`, and `docs/widget-embed.md`.
- **Token storage helper (`src/widget/storage.ts`) runs ONLY in the iframe.** It namespaces keys by public key (`ar-support:session:<key>`, `ar-support:token:<key>`, `ar-support:conversation:<key>`), mints/reuses a stable per-(browser, public key) session id on `init` (client continuity/reconnect groundwork - it is NOT transmitted on any request and is NOT the per-session rate-limit key; US-076 keys its per-session window on the XFF-derived client IP via `_widget_client_ip`, not this value) and holds the token + conversation-id slots US-084 populates (the token from the US-078 `X-Conversation-Token` response header; the conversation id stored alongside it so a reload can resume the transcript - the id is not a credential, the token↔id binding is enforced server-side, US-071). All best-effort (private-mode safe).

### In-iframe chat UI + theming via loader `init` (US-084, ADR-0008)

US-084 renders the chat UI INSIDE the US-083 iframe and themes it ENTIRELY from the loader's `init` config (no host CSS can reach a cross-origin iframe). `frontend/src/widget/api.ts` is the anonymous public-surface client; `useConversation.ts` is the chat state machine; `WidgetApp.tsx` renders launcher + panel (message list, composer). **Frontend-only EXCEPT one tiny US-074 CORS addition** (below); no migration, no new env.

- **KNOWN cross-cutting integration concern (surfaced, NOT introduced by US-084):** this implementation defers ALL widget-key resolution to the FIRST message (US-078 re-resolve, sent from the iframe = kit origin); the loader does NOT call `POST /widget/keys/resolve` - it only injects the iframe and relays postMessage, and the iframe only fetches `/widget/conversations/messages` (on send) and the transcript (poll). ### Customer SSE channel + multi-instance LISTEN/NOTIFY fan-out (US-081, ADR-0008)

US-081 is the customer leg's LIVE PUSH - the long-lived backend SSE over which async AGENT replies (US-082) reach the anonymous customer, plus the OPTIONAL cross-instance transport. It completes the seam US-082 left: US-082 shipped the single-instance `ConversationFanout` registry (the `publish`/`subscribe` primitives); US-081 adds the SUBSCRIBER (the customer GET SSE) and the multi-instance bridge that feeds INTO that same registry. **No new migration, no new table** (`LISTEN`/`NOTIFY` are built-ins); the one new env is the OPTIONAL `WIDGET_FANOUT_DATABASE_URL`. Same seam-before-call-site split as US-075→076 / US-077→079 / US-082→081.

- **`GET /widget/conversations/{id}/events` (`main.py:widget_conversation_events`) is the customer SSE - authed by the US-071 OPAQUE token, NEVER a Supabase JWT.** This is what keeps the anonymous customer structurally OFF the Supabase Realtime/JWT surface (ADR-0008 max-isolation): a plain backend SSE + opaque token, no supabase-js channel on the customer leg. `EventSource` can't set headers, so the iframe consumes it via fetch + ReadableStream (like the POST stream) to keep the token in the `X-Conversation-Token` HEADER, never a URL/log. It resolves the token with `slide=False` (holding a passive read channel open is NOT activity that should slide the 24h window - the US-071 transcript-read precedent) and enforces the path-id BINDING (`conv["id"] == conversation_id`), so a token for X can never open Y's stream; a missing/expired/resolved/mismatched token → opaque 401. It then drains `_CONVERSATION_FANOUT.subscribe(conversation_id)`, emitting `event: ready` then one `event: message` per reply (the customer-safe body only - id/role/content/created_at). Per-session rate-limited at open (US-076; no `public_key` on this path) as defense-in-depth against SSE-open spam. NOT a `_WIDGET_AUTHENTICATED_SUFFIXES` route, so US-074's PUBLIC-widget CORS owns it (the anonymous surface).
- **Liveness + close-on-invalidation.** A `: keepalive` SSE comment every `WIDGET_SSE_KEEPALIVE_SECONDS` (25) survives proxy idle-timeouts; the token is re-validated every `WIDGET_SSE_REVALIDATE_SECONDS` (60, slide=False) and a now-invalid token (the conversation was RESOLVED → its token purged, US-071) emits `event: close` and ends - so the connection closes when the token is invalidated (the AC). The stream uses a PERSISTENT get-task across keepalive timeouts (`asyncio.wait` leaves the pending `queue.get()` task alive rather than cancelling it), so a reply arriving exactly at a heartbeat boundary is never dropped between iterations. Forward-looking: a future resolve path can `publish` a `{"type":"close"}` event for INSTANT teardown (US-086+); the generator already handles it.
- **`backend/conversation_bridge.py:ConversationBridge` is the OPTIONAL multi-instance transport - Postgres `LISTEN`/`NOTIFY`, no Redis/queue infra.** Built + started at startup ONLY when `WIDGET_FANOUT_DATABASE_URL` (a DIRECT asyncpg DSN for the shared Postgres - LISTEN needs a real connection PostgREST cannot hold) is set; unset → single-instance fan-out only (the common kit deployment) and the bridge is never built. After the agent-reply endpoint (US-082) durably writes + `publish()`es LOCALLY, it ALSO calls `bridge.notify(conversation_id, event)` (best-effort, non-fatal), emitting a single-channel (`widget_conversation_events`) `NOTIFY` carrying `{origin: <this instance's id>, conversation_id, event}`. Every instance runs a supervised `LISTEN` connection (reconnect with backoff + a `SELECT 1` health probe so a silently-dropped socket self-heals) and `publish()`es each notification LOCALLY - EXCEPT a frame stamped with its OWN `instance_id`, which it IGNORES. That self-dedup is load-bearing: the originating instance already delivered via its local `publish()`, so replaying its own NOTIFY would double-push. `publish` no-ops for conversations an instance holds no subscriber for, so one shared channel + one background LISTEN task is correct and cheap (per-conversation channels are a noted future refinement, unwarranted at support-reply volume).
- **Fail-soft everywhere - the bridge is never a request-path or startup SPOF.** A DB blip at startup leaves the NOTIFY pool unset (lazily self-heals on first use) and the LISTEN supervisor retries; `notify` swallows transport errors; a malformed/foreign NOTIFY frame is dropped, never raised. A NOTIFY payload over `_MAX_NOTIFY_PAYLOAD_BYTES` (7000, under Postgres's 8000 cap) SKIPS the cross-instance nudge (the remote customer recovers via the transcript) - the local `publish` always delivers in full. The worst case is a degrade to single-instance delivery + the durable-transcript backstop. `workspace_membership.role` appears nowhere (ADR-0002 intact); the bridge is pure transport and assigns the conversation id no meaning.
- **Frontend (`api.ts:subscribeConversationEvents` + a `useConversation.ts` effect).** A fetch-based SSE with the token header parses `ready`/`message`/`close`; the effect holds the stream open whenever a conversation + token exist, reconnecting on drop, and feeds the SAME `refreshTranscript` seam US-084 named - a `message` is a low-latency NUDGE to re-read the durable transcript (the transcript stays the source of truth, so cross-instance content size never matters client-side). The interim transcript POLL is RETAINED as a backstop, so a dropped/closed SSE never loses a reply; the SSE is purely a latency win.

### Session-scoped retrieval memory: transcript ≠ retrievable (US-085, ADR-0004/0008)

US-085 is a SECURITY-PROPERTY story that pins an invariant already true structurally - it ships NO behaviour change, NO migration, NO env, NO new call site: only a regression test (`backend/test_us085_session_scoped_retrieval.py`) + this note.
The property (CONTEXT "two stores, one word"): the support bot's *retrievable/answerable* surface is **session-scoped** - a customer's typed input NEVER becomes a retrievable `chunks` row and never bleeds across sessions - and is **DISTINCT from** the durable, agent-readable transcript (`conversation_messages`), which is written for the operator but is NEVER a retrieval source.
The two stores share the word "message" but are different trust objects: the transcript is *readable* (by a workspace member via US-071/082), the corpus is *answerable* (by anyone who can reach the widget, via the bot's synthesized faithful answer). Keeping customer input out of the corpus is what stops a pasted secret from becoming answerable to the next anonymous visitor.

- **The retrievable corpus (`chunks`) has exactly ONE writer: authenticated document ingestion.** `main.py:_insert_chunk_rows` / `_reconcile_chunks` require an `AuthedUser` JWT + a real `document_id` from a parsed document, and are reached ONLY from the ingest path - never from any `/widget/*` route. The customer-message path (`widget_conversation_message` → `_persist_conversation_message`, US-078/079) writes ONLY `conversation_messages` under the service role; that table carries none of the columns that make a row retrievable (`embedding` / `content_hash` / `chunk_index` / `document_id`). There is **no code path** from customer-pasted text to a `chunks` row or a `chunk_acl` grant (AC3) - share-to-bot (US-086) is a deliberate doc-owner publish action on an ingested doc, never a customer-input write.
- **The bot's retrieval reads `chunks` ONLY, through `match_chunks`/`keyword_search`, and NEVER the transcript.** `run_bot_deflection_turn` → `run_deflection_pipeline` → `hybrid_search` touches exactly the two RPCs (`/rest/v1/rpc/match_chunks`, `/rest/v1/rpc/keyword_search`) under the bot's minted JWT (US-070); `conversation_messages` appears nowhere in the retrieval path, so a prior session's transcript is structurally unreachable as a retrieval source. The customer message is CONSUMED as the search *query* (it rides `keyword_search`'s `query` field, and its *embedding* - never the raw text - rides `match_chunks`' `query_embedding`); using input as a query is not the same as inserting it into the corpus, and there is no insert.
- **No cross-session bleed is a consequence of the two facts above, not a separate mechanism.** Session A's paste lands only in A's transcript (never `chunks`); session B's retrieval reads only `chunks` (never any transcript); therefore B cannot surface A's paste regardless of content. The bot's per-turn JWT + workspace-membership RLS (US-070) already scope even the corpus to what the bot may see; US-085 pins the orthogonal axis that customer input never enters that corpus in the first place.

### Optional UNENFORCED soft-claim: advisory dim, never a gate (US-089, ADR-0004/0008)

US-089 adds the OPTIONAL soft-claim on the US-087 queue: an agent claims an escalated conversation so the row dims for other agents, cutting accidental double-replies.
It is ADVISORY ONLY and DELIBERATELY UNENFORCED - last-write-wins, and it NEVER gates the reply or resolve actions (the PRD failure indicator is "claim blocks A2's reply", which would make it an enforced assignment - explicitly out of scope, S5 routing stays deferred).
**Frontend-only: NO new migration, NO new backend, NO new env, NO new table.**
The `claimed_by`/`claimed_at` columns and the member-UPDATE RLS already exist from US-066; the claim write rides the SAME `conversations_update_member` membership policy US-088's Resolve rides.

- **The claim is a plain member UPDATE under the agent's own JWT, byte-identical posture to Resolve.**
`supportQueue.ts:claimConversation` PATCHes `claimed_by=<caller uid>`, `claimed_at=<client clock>` (the DB does NOT pin `claimed_by=auth.uid()` - the value is honest, not a trust boundary, since the claim is advisory); `releaseConversation` clears both.
`role` enters NO predicate (membership-gated like the whole queue, ADR-0002 intact); a non-member's claim affects **zero** rows (RLS), and a claim can never cross the tenant boundary.

- **The reply path has ZERO claim check, which is what makes "unenforced" structural.** The agent-reply endpoint (US-082) and the customer message turn (US-079) never read `claimed_by`; a claimed conversation is replyable by any workspace member exactly as an unclaimed one is.
This is the failure-indicator-is-absent property, verified live: a second member posts an `assistant` reply to a CLAIMED conversation and gets 201.

### `/support/settings` admin route: keys, enable-support, share-to-bot (US-090, ADR-0002/0008)

`/support/settings` is the ADMIN surface - the one place `role='admin'` legitimately gates a UI (ADR-0002: administering widget keys IS the administrative action the admin role exists for), the deliberate contrast to the MEMBERSHIP-gated `/support/queue` (US-087). **Frontend-only: NO new migration, NO new backend, NO new env.** It is a thin admin client over the ALREADY-SHIPPED US-072 widget-key endpoints (`/api/support/widget-keys*`, whose docstrings already name US-090 as their caller) and the US-086 share-to-bot endpoints. `frontend/src/lib/supportSettings.ts` is the widget-key client; `frontend/src/pages/SupportSettingsPage.tsx` is the page; the route lands in `App.tsx` under `ProtectedRoute` and an admin-conditional `AppHeader` "Support settings" nav link.

### User-initiated "talk to a human" widget button (US-091, ADR-0003/0004/0008)

US-091 is the explicit "talk to a human" control in the customer widget: a deterministic, UI-INITIATED escalation, NEVER a model `escalate()` tool (ADR-0003 - escalate-vs-answer is deterministic control flow, not something the model chooses).
**Frontend-only: NO new migration, NO new backend, NO new env.** The backend latch endpoint (`POST /widget/conversations/escalate`) ALREADY shipped in US-080 (its docstring names US-091 as the caller); US-091 is just the widget button that calls it.
`frontend/src/widget/api.ts:escalateConversation` is the client; `useConversation.ts` gains the `escalate` action + a reactive `hasConversation` flag; `WidgetApp.tsx` renders the button at the top of the composer footer; `widget.css` styles it as a subtle secondary control.

- **The button calls the SAME US-080 latch path the model-mediated decision uses**, authed by the US-071 opaque `X-Conversation-Token` (NOT a Supabase JWT - the anonymous customer is off the Supabase trust surface). On success the conversation latches `status='escalated'` (the US-067 trigger owns `escalated_at`, set-once, so deflection stays derivable), the bot goes silent thereafter, and every later customer message routes to the human queue with no bot answer (the US-080 routing in `widget_conversation_message` short-circuits an already-`escalated` conversation to `conversation` + `done`). It runs NO pipeline (UI-initiated, never a model tool) and is per-session rate-limited (US-076).
- **The control is offered ONLY for an ACTIVE conversation that already exists on the server** (`canEscalate = status === 'active' && hasConversation`). `hasConversation` is a reactive mirror of "a conversation (+opaque token) exists" - set true on the first-message `conversation` event or a resume, cleared alongside the token whenever it dies (401 in send/transcript/SSE). The endpoint needs a token to resolve its ONE bound conversation, so the button never appears before the first message and never tries to escalate a non-existent conversation; a customer with no conversation yet simply types a message first (the natural flow). NO confirmation dialog - "I want a person" is a benign, expected customer action, and friction there is bad UX (the contrast with US-086's share-to-bot, which IS a security-sensitive publish that must be confirmed).

The composer STAYS enabled (escalated is not a disabled state - only `sending`/`throttled`/`resolved` disable it), so later messages still send and route to the queue; the human reply arrives asynchronously over the US-081 customer SSE. Escalating is idempotent server-side (`escalated -> escalated` is a US-067 no-op).
- **Fail-safe error handling mirrors the send path.** A 401 (token expired/resolved before the click) clears the dead conversation and surfaces "send a message to start a new one" rather than dead-ending the customer; a 429 (US-076 per-session throttle) and a 502 (the strict US-080 latch write failed) surface a retry hint via the existing `error` banner. `escalating` disables the button during the in-flight call to prevent double-submit.
