The RAG kit that shows its work.

Purvia is an MIT-licensed, permissions-aware RAG kit. Chunk-level ACLs are enforced inside the retrieval SQL predicate and checked by zero-leak evals in CI. A deterministic escalation gate keeps the support bot under a false-resolve ceiling you set. Raw OpenAI SDK + Pydantic; no orchestration framework in the product. Every claim on this page ends in a file path you can open.

Permissions live inside the retrieval predicate

The naive approach post-filters: retrieve top-k, then drop what the viewer cannot see. At 5% visibility and top-10, the expected number of visible chunks in the result is k times selectivity: 10 x 0.05 = 0.5 chunks. Purvia pushes owner-OR-ACL and workspace membership into the SQL the planner executes, under the viewer's JWT (SECURITY INVOKER), so the database ranks only what the viewer may see. A forgotten backend filter cannot cross the tenant boundary, because there is no backend-passed tenant id to forget.

The actual predicate 
where c.embedding is not null
  and d.deleted_at is null
  and (1 - (c.embedding <=> query_embedding)) >= match_threshold
  and (c.user_id = auth.uid() or ca.chunk_id is not null)   -- owner OR chunk-level ACL
  and exists (
    select 1
    from public.workspace_membership wm
    where wm.workspace_id = d.workspace_id
      and wm.user_id = auth.uid()                           -- tenant boundary, from the JWT
  )

Attacked, not just asserted
  • E6 seeds a second workspace with an identical corpus, grants the viewer ACLs on both but membership in one, and asserts recall@10 == 0.0 across the boundary, with a positive control. Fails the required CI check on any leak.
  • AU4 throws forged, missing, and expired JWTs, a cross-workspace JWT holding an ACL on the other tenant, and revoked widget keys at the API edge. Over 30 exact assertions (== 0 rows or 401).
  • No security verdict has a config off-switch: the gate loader rejects any downgrade at load time, so burying a leak finding requires a code-visible PR diff. Precise scope: an E6 run that cannot execute at all (transient infra failure, after retries) is flagged loudly but deliberately does not fail the check; that trade-off is documented in the runner and in the honest section below.

The support bot's risk tolerance is a number

A support bot that confidently answers what it should have escalated is a liability with a court record. In Purvia, escalate-vs-answer is deterministic control flow. The model drafts; two gates decide. There is no model escalate() tool.

The pipeline, from the docstring that ships 
retrieve (hybrid, ONCE) → retrieval gate
    → weak  ⇒ escalate now (ZERO draft, ZERO judge calls)
    → strong ⇒ draft → faithfulness gate
        → faithful   ⇒ answer (send the draft)
        → unfaithful ⇒ escalate
backend/escalation.py (run_deflection_pipeline)
  • One buyer-set risk number. ESCALATION_FALSE_RESOLVE_CEILING, default 5%: the maximum fraction of should-escalate questions allowed to auto-resolve. Enforced by the weekly eval gate, never the request path, so your risk tolerance costs zero latency. A breach fails the workflow and files an issue; it is never downgraded to a comment.
  • Fails closed everywhere it matters. A judge error, timeout, or refusal escalates rather than auto-sending. Without the service-role key, every widget route returns 503. The escalation message a no-access customer sees is asserted byte-identical to the ordinary deferral, so escalating never discloses that restricted content exists.
  • Deflection is derived, not stored. resolved with escalated_at null means the bot handled it alone; the timestamp is stamped by a Postgres trigger that callers cannot fake. No resolved_by_bot column to game.
  • Caveats, stated first: the gate thresholds are documented placeholder values, not tuned; the published LLM-judged history is thin (one RAGAS weekly snapshot; no escalation-weekly reports committed yet). The gate infrastructure is in CI; its published track record is short.

Determinism decides CI placement, structurally: declaring an LLM-judged gate as a per-PR fail-on-red is a load error (evals/gate/placement.py). A judge wobble cannot red-bar a merge.

  • 60-question golden set across five categories, including adversarial and exact-token lexical questions. Small on purpose; the eval doc's own limitations section says the per-category cells have low statistical power. And to be precise about what the nightly series is: the same deterministic assertions rerun against an unchanged corpus, a change-detector and an audit trail, not 38 independent trials.
  • Cross-family judging: OpenAI writes answers, a Claude model grades them. The Anthropic key is eval-only and never read by the live backend.
  • Golden sets are built to fail loud on a corpus swap: content anchors are quoted verbatim from the demo corpus, so replacing the corpus and authoring your own golden set are the same step, by design.
  • Reranker decision made with data and published: LLM rerank wins adversarial MRR by +0.052 but costs ~1.2s median per query, so the default stays none (docs/reranker-bakeoff.md, source JSONs committed).

What this does not do yet

This list is maintained with the same discipline as the eval tables. If a claim above contradicts something here, the repo wins and I fix the claim.

  • The 10k-chunk permissions scale benchmark is currently broken. Every nightly report since 2026-06-19 publishes 0.000 in every cell; the likely cause is that the workspace-membership migrations orphaned the benchmark's synthetic viewers. The regression alarm that is supposed to catch exactly this has been firing into a void. Fixing it is a precondition for taking the scale story seriously; until then the HNSW findings in the README are design-time results, dated June 2026. docs/permissions-scale-nightly/
  • Hybrid MRR still trails vector on paraphrase questions (0.767 vs 1.000 in the 07-15 nightly). The fix PRD's own goal is not fully met on that cell.
  • The demo corpus is 8 documents and 16 chunks, and the receipts do not transfer to your corpus: swapping in your documents and authoring your own golden set are the same step, by design. Day one on your data starts at zero receipts plus a labeling contract where under-labeling is a security defect. A semi-automated golden-set bootstrapper is the roadmap answer; it is not shipped.
  • The retrieval nightly itself went down for 18 nights in June (it ran and failed at its seed step, 06-01 to 06-18, per the Actions history) and I did not notice. Same lesson as the dead leg, and I only found it during this launch audit.
  • The E6 security gate fails open on infrastructure failure: a run that cannot execute after retries is flagged loudly but does not fail the check. A leak verdict cannot be downgraded; a flaky day can slip a run. Making repeated execution failure fail-closed is an open decision.
  • Adoption assumptions, stated plainly: Supabase is the identity and RLS substrate (auth.uid() from GoTrue JWTs is load-bearing; bringing your own auth means porting the predicate, RLS, JWT minting, and eval seeds), and the default install pulls docling, which brings torch. Parsing sits behind a seam, so an optional-extra split is feasible; it is a roadmap item.
  • Zero users, zero revenue, zero testimonials. Not deployed to a public URL. You are early, which is the honest word for it.
  • Bus factor is one. If I take a job tomorrow, what survives is what you already have: MIT-licensed code and the eval harness that lets a fork prove it still holds its properties. That is the whole succession plan, and I would rather say so than let you find out.
  • Solo repo: PRs are self-merged, and main has no branch protection yet, so "fails the required CI check" is a convention I honor, not a GitHub-enforced block. Enabling a ruleset is on the pre-launch list.
  • One RAGAS weekly snapshot published; no escalation-weekly history yet. The gates exist; their long-run record does not, yet.

Pricing

Core - free, MIT, forever
  • Everything described on this page: the predicate, the ACLs, the widget, the escalation gate, the full eval harness, all of it already in the repo under an irrevocable MIT grant.
  • A receipt behind a paywall is not a receipt. The proof stays free because the proof is the pitch.
Pro - planned, unpriced
  • Stated plainly: a pro tier would gate only what does not exist yet - future hardening modules, a maintained update stream, priority support. Nothing already MIT will ever move behind it, and there is no price tag here because there is nothing to sell yet.
  • The working hypothesis is a one-time price in the developer-asset band; it gets tested in buyer conversations, not on this page.

Who built this

I spent about ten years in hardware validation and two in applied AI consulting. Hardware taught me that a system you have not tried to break is a system you know nothing about. Purvia is that discipline applied to RAG: golden sets, non-regression baselines, one-sided alarms, fail-closed defaults, and a public record of the times it failed. I have not proven this sells. I have proven it leaks zero labeled chunks across the 180 no-access runs in every published nightly, and you can rerun that sentence. One more receipt about how it was built: 49 of the 152 human-authored commits on main say Co-authored-by: Claude. The workflow is PRD-driven agent development with the deterministic eval gates as the reviewer that never sleeps; the git log is the disclosure.
