1. Device description

PURVIA-1 is a self-hosted retrieval-augmented generation stack: FastAPI backend on the raw OpenAI SDK + Pydantic, Postgres/pgvector via Supabase, React frontend, and an embeddable cross-origin support widget. Access control is evaluated inside the retrieval SQL predicate (owner-OR-ACL AND workspace membership, resolved from the caller's JWT under SECURITY INVOKER). An eval harness with 6 CI workflows checks isolation, retrieval quality, and escalation safety; deterministic security gates fail their CI check on any breach. No orchestration framework in the product (the LangSmith tracing SDK is a runtime dependency; RAGAS/langchain are lazy eval-only extras).

5. Verification methodology

  • 6 CI workflows; determinism decides placement: only deterministic gates (E6, E7 P1a/P1b, AU4, ship-green) may fail a PR/push check; LLM-judged gates run weekly and file issues. Declaring an LLM-judged gate per-PR is a structural load error (evals/gate/placement.py).
  • Cross-family judging: OpenAI generates, claude-haiku-4-5 grades (evals/retrieval/runner.py). The judge is a fixed instrument, excluded from the configurable model surface.
  • Non-regression baselines are one-sided (US-118): improvements never flag. Adopted after the two-sided predecessor spent the dead-leg period in permanent false alarm.
  • 38 nightly snapshots committed (2026-05-19 to 2026-07-15, with gaps); bad snapshots are retained, not deleted.
  • AU4: 30+ exact API-edge assertions (forged/missing/expired JWTs, cross-tenant ACL holders, revoked keys, opaque-token-as-JWT), its own per-PR job.
