# Agentic RAG — Context glossary

Terms that have a precise meaning in this codebase. Resolved during design sessions; update inline as decisions are made.

## Permissions / access control (Module 11, in design)

- **Owner** — the user identified by `chunks.user_id` / `documents.user_id`. Set at ingestion time; never null. Always has access to the row regardless of [[chunk-acl]] state. Distinct from "viewer".
- **Principal-set resolution** — happens inside `match_chunks` (SECURITY INVOKER) using `auth.uid()`. The function looks up `principal_membership` itself; callers do **not** pass `principal_ids`. The trust boundary is the database, not the backend. The query-time predicate is `c.user_id = auth.uid() OR EXISTS (chunk_acl row …)` — owner OR ACL. Phase 2 AND-s a workspace-membership clause onto this (ADR-0002): the effective predicate is `(owner OR ACL) AND viewer ∈ members(document.workspace_id)` — see *Workspace tenant isolation* below.
- **Viewer** — the authenticated user issuing a retrieval query. May or may not be the owner. A viewer has access to a chunk iff they are the owner OR a principal in their resolved set has an entry in [[chunk-acl]].
- **chunk-acl** — the `chunk_acl(chunk_id, principal_type, principal_id)` table. **Sole source of truth** for non-owner access. Holds *additive* grants on top of ownership; an empty ACL list does **not** mean "no one can see this row" — it means "only the owner can." Existing chunks need no backfill when this table is introduced. There is no `document_acl` table — doc-level grants are an *operation* the backend performs by inserting one row per chunk in a transaction. Overrides are performed by deleting the inherited rows for the affected chunks and inserting the override rows in their place.
- **Principal** — any entity that can appear on the right-hand side of a grant. In v0, principal types are `user` and `group`. A `user` is a row in `auth.users`. A `group` is a named bag of users with no other semantics. Roles (in the RBAC sense) are deferred — if needed later, they layer on as a separate permission-bundle table orthogonal to the ACL.
- **Group membership** — `principal_membership(principal_id, member_user_id)`. Maps a group to its member users — flat only in v0. A group cannot contain another group. Forward-compatible: a future `member_type` column can add `'group'` and a recursive CTE without breaking existing rows. Production deployments typically get the *flattened* membership from an IdP (Okta SCIM, Google Directory); deferring nesting until an IdP integration exists is consistent with how real systems are built. Single-namespace (no workspace/tenant scoping) was a v0 caveat for production credibility — closed in Phase 2 by the **Workspace** tenant boundary (ADR-0002; see *Workspace tenant isolation* below). Group nesting itself remains deferred.
- **Doc-level share** — there is no persisted `document_acl` table. "Share this doc with finance" is a backend operation that opens a transaction and inserts one `chunk_acl` row per chunk in the document. Doc-level intent is *not stored* — it can be reconstructed by aggregating `chunk_acl` ("every chunk in doc X grants finance" → "doc X is shared with finance"). The UI's share dialog uses this aggregation to render current state.
- **Re-chunking caveat** — because `chunk_acl` is the sole source of truth and grants are denormalized onto chunk UUIDs, any operation that re-chunks a document (different chunk size, chunker-version bump, content edit) loses all grants on the destroyed chunks. The operator (or a re-chunk handler we add later) must snapshot the doc-level intent from the old chunks before re-chunking and replay it onto the new ones. Document this as a known limitation in `docs/permissions-aware-rag.md`.

## Workspace tenant isolation (Phase 2, ADR-0002)

- **Workspace** — the hard tenant boundary layered *above* owner-OR-ACL. A `workspaces(id, name)` registry row; creation is operator-level in v1 (seed script / admin endpoint), never self-serve. Existing data migrates into one fixed-UUID **Default Workspace** (`00000000-0000-0000-0000-0000000000d0`) so the boundary is a no-op for the legacy corpus — it only bites once a *second* workspace exists.
- **Workspace membership** — `workspace_membership(workspace_id, user_id, role)`, the load-bearing join the boundary resolves against. A chunk is visible iff `(owner OR ACL grant) AND viewer ∈ members(document.workspace_id)` — the membership clause is an `EXISTS` against this table keyed on `auth.uid()`, resolved *inside* `match_chunks` / `keyword_search` (SECURITY INVOKER) and mirrored into the `chunks` / `documents` SELECT RLS via the SECURITY DEFINER helper `_user_in_document_workspace` (which breaks the policy-recursion cycle, like `_chunk_acl_grants_user`). The backend never passes a workspace id as the boundary (the ADR-0002 rejected alternative) — a forgotten backend filter can only *widen* within the viewer's own workspaces, never leak across them.
- **Subtractive boundary** — the membership clause is AND-ed under owner-OR-ACL, not OR-ed, so it can only *remove* visibility: even a document's owner sees nothing if they are not a member of its workspace (intentional). Unlike additive [[chunk-acl]] there is no safe "do nothing" default, so the US-002 migration *actively* places every existing user + document into the Default Workspace. `documents.workspace_id` is NOT NULL with a transitional default of the Default Workspace (US-007 stamps the resolved active workspace explicitly and may drop the default).
- **Membership `role`** — `role ∈ {admin, member}` is *administrative only*: it governs member/group management and grants **no** content access, so it never appears in any retrieval predicate. A different axis from the deferred access-control (RBAC) role concept; an admin who must read content is granted it via the ACL like anyone else.
- **Workspace-scoped Principals** — `principals` became workspace-local in US-006: a `workspace_id` column, uniqueness moved from global `name` to per-workspace `(workspace_id, name)`, and SELECT RLS tightened from `using (true)` to membership-gated. This is leakage/collision hygiene (two workspaces can each own a "finance" group; one workspace can't enumerate another's group names), **not** the load-bearing access control — the content boundary is the membership clause above, which blocks cross-workspace retrieval even on a mis-grant. Because `_resolve_principal` (`backend/main.py`) reads `principals` under the caller's JWT, out-of-workspace groups resolve to nothing (404) automatically; resolution is additionally scoped to the target document's workspace so the now-non-unique `name` binds deterministically under `limit 1`.
- **Active workspace** — a *non-security* UX filter (which workspace a query/upload acts in), carried in the API path and validated against the caller's memberships (default-when-sole, 400-on-ambiguous). Deliberately not trust-load-bearing because the membership clause already prevents leakage. Deferred to US-007/US-008 (not part of US-001–US-006); the production signup → Default-Workspace membership step is deferred with it, so seed/eval/test users get their memberships from the Python seeders + share-API fixtures. Its retrieval-side mechanism shipped early in US-070 as the optional `filter_workspace_id` param on `match_chunks` / `keyword_search` (`20260624150000` / `20260624150100`) - a pure narrowing filter AND-ed beside `filter_topics`, `null`-default no-op, subtractive-only, never the boundary. The first consumer is the support-bot retrieval turn (it passes the conversation's workspace); the authenticated `/api/chat` path still passes none, so E4/E6 stay byte-identical and the active-workspace *resolution* stays deferred to US-007.

## Model surface (Phase 2, ADR-0006)

- **Chat mode** — the runtime answerer surface. `completions` (OpenAI Chat Completions) is the **portable** path every provider speaks and the cross-provider default; `responses` (OpenAI Responses API) is an **OpenAI-proper-only** enhancement layering hosted `file_search` (server-side vector store) + server-side `previous_response_id` threading. **Responses-mode runs on OpenAI proper only (`provider=openai` with no `base_url` override), non-portable** — the Responses endpoint doesn't exist on Azure or any OpenAI-compatible `base_url` host (vLLM/Together/Groq/…), so an `openai` answerer with `OPENAI_BASE_URL` set is *not* responses-capable (F3 capability-matrix row; full matrix + env reference + worked Azure example in [[model-surface-doc]] = `docs/model-surface.md`, US-028).
- **Fail-closed chat-mode binding (US-025, FR-M4)** — the default chat mode is resolved against the already-validated answerer config (provider AND `base_url`, via `responses_capable` / `resolve_chat_mode_default`, `model_config.py`), once at startup: a non-responses-capable answerer (Azure, or an `openai` answerer with a `base_url` override) with `CHAT_MODE_DEFAULT=responses` **raises `RuntimeError` and refuses to start** (never a silent downgrade that would strip file_search/threading while looking "accepted"); with the var unset it resolves to `completions`. For an OpenAI-proper answerer (no `base_url`) the historical US-004 `responses` default is preserved. The same gate covers the per-request `mode` override: `mode=responses` on a non-responses-capable answerer fails closed with a 400 at `/api/chat`, and `/api/config` only advertises `responses` in `supported_chat_modes` when the answerer is responses-capable. So the hosted-`file_search` (`OPENAI_VECTOR_STORE_ID`) + `previous_response_id` threading path is reachable only on OpenAI proper.
- **Embedding stamp (`embedding_config`, US-026)** — a **single-row** table (`model`, `dim`, `indexed_at`; `singleton boolean` PK pinned TRUE by a CHECK) recording what the corpus was embedded under — **one model per corpus, not per chunk**. Retrieval only works when the query and stored chunk vectors come from the *same* model; mixing models degrades recall with no error (the dangerous same-dims-different-model case, e.g. `text-embedding-3-small` vs `text-embedding-ada-002`, both 1536). The DB already knows the dimension (baked into `chunks.embedding`'s `vector(1536)`, which stays the source of truth for `dim`) but not the model name; the stamp records both so a later embedder change is *detectable* (US-027's startup probe-embed compares config vs stamp and refuses to start on a mismatch) rather than silently rotting retrieval.
- **Stamp write paths (service-role only, by design)** — the `embedding_config` table has **no INSERT/UPDATE/DELETE policy for the API role** (only an authenticated SELECT), so every write goes through the **service-role** key. The production ingest endpoint is **insert-if-absent** (`_stamp_embedding_config`, `ON CONFLICT DO NOTHING`, now writing with `_service_role_headers()` — it skips, logged, when `SUPABASE_SERVICE_ROLE_KEY` is unset, matching the US-027 guard which also disables itself then): the first ingest that produces embeddings seeds the row, and a routine ingest never silently rewrites the recorded model (that would blind the drift guard). Restricting writes to service-role also closes the cross-tenant poisoning hole the old `with check (true)` authenticated-insert policy opened (any tenant could pre-seed the global singleton and trip the guard for everyone). Bulk re-index — the corpus + wikipedia seeders, which embed via direct asyncpg inserts and so must stamp themselves (a freshly-seeded corpus would otherwise have chunks but no stamp) — **overwrites** the row (`stamp_embedding_config`, `do update`), running as service-role. The stamped `dim` is the *actually-produced* vector length; pgvector rejects a wrong-length insert, so by stamp time it necessarily equals the column dim.
- **Re-embedding vs re-chunking** — the US-027 re-index remedy is a **re-*embed*** (recompute vectors for the *same* chunks under a new model/provider, optionally at a migrated dim). Re-embedding **preserves chunk UUIDs**, so the [[chunk-acl]] grants keyed on those UUIDs survive — the safe operation, plus a `chunks.embedding` column migration only when the dimension itself changes. This is the opposite of **re-*chunking*** (different chunk size / content edit), which destroys the chunk UUIDs and with them every grant (the *re-chunking caveat* under *Permissions*). State this distinction wherever the re-index remedy appears.
- **Operator reference (US-028)** — `docs/model-surface.md` is the canonical how-to for configuring your own model host: every env var (per-role provider/connection + per-call-site model selectors) with the role-fallback precedence, a worked Azure example (deployment-vs-model split, `/healthz` verification), the **F3 capability matrix** (tested `openai`/`azure`; OpenAI-compatible `base_url`s supported-but-untested; native non-OpenAI runtime APIs out; Responses-mode OpenAI-only; Entra/AAD auth deferred), and the embedder re-index procedure kept in sync with the US-027 guard error. README "Environment variables → Model surface" is the short index into it.

## Ingestion boundary (I5 / I6, ADR-0007)

- **Single-module docling seam (I5 = verified, US-036)** — the document parser lives behind one clean seam: `docling` is imported in **exactly one** module (`backend/parsing.py`), whose single entry `parse_document(raw, filename, content_type) -> str` (`backend/parsing.py:439`) returns normalized markdown. No other `backend/*.py` imports docling or names a docling-only type, and `chunk_text(text: str, ...)` (`backend/chunking.py:109`) consumes only `str` with zero docling awareness. The PRD listed I5 as "PARTIAL — refactor if docling types are threaded"; docling was never threaded, so I5 is **verified, not refactored** — and that "PARTIAL" qualifier is moot. The verification is a standing invariant enforced by `backend/test_parsing_seam.py` (a future leak — e.g. importing a docling type into `main.py` — breaks the test), not a one-time assertion.
- **Markdown-string output contract (v1, pinned)** — the boundary emits a normalized markdown `str`, the load-bearing coupling to the chunker (which is markdown/heading-aware only). A parser with native structured output (tables/layout/bboxes) MUST **flatten to markdown at the boundary**. Widening the output type to carry structure for a future *layout-aware* chunker is a documented **future change (US-045)** that also touches `chunking.py` — explicitly **not v1** (a deliberate pin with a known upgrade path, not an oversight). v1 code never carries a non-`str` document type across the seam (enforced by `backend/test_parsing_seam.py`, US-036).
- **Swappable parser (I6)** — the implicit function seam is lifted into a one-method `DocumentParser` protocol + `build_parser` factory mirroring the existing `Reranker` (`backend/reranking.py`) / `WebSearchProvider` (`backend/web_search.py`) factory patterns, selected by `PARSER=docling|unstructured|llamaparse`. Default `docling` preserves today's behavior; commercial adapters fail at **build time** on a missing key; an unimplemented value fails loudly — **never a silent fallback to docling**. The `pypdfium2` PDF fallback stays **inside** the docling adapter, not promoted to the boundary. One real commercial adapter ships — **LlamaParse** (simplest API + strongest OCR/complex-layout coverage, the gap docling's default text pipeline can't cover, PRD §3.3); **Unstructured.io** is the named "bring your own adapter" example. Adding a parser is a supported path documented in the **"write your own adapter" guide** (`docs/ingestion-parser-adapters.md`, US-043): subclass `DocumentParser` → add the name to the `PARSER` validation (`ParserName` Literal + `get_parser_name` tuple) → register in `build_parser` → select via `PARSER`, with the markdown-string flatten rule pinned and the US-041 smoke test as the round-trip proof template. Accepted gaps (OCR-not-default, markdown-string ceiling, unstructured-is-BYO) are F3 capability-matrix rows citing ADR-0007 — the ingestion **F3 capability matrix** lives in `docs/ingestion-parser-adapters.md` § "F3 capability matrix" (US-044), the sibling of the model-surface matrix in `docs/model-surface.md`.

## Escalation signal & deflection pipeline (Phase 2, ADR-0003)

- **Deflection pipeline** - the support-face answer-vs-escalate decision, run as a deterministic **retrieval-first cascade**: `retrieve (hybrid, once) → retrieval gate → [if strong] draft → faithfulness gate → answer-or-escalate` (`run_deflection_pipeline`, `backend/escalation.py`, US-049). It is **plain control flow - never a model `escalate()` tool, never the M1 agentic loop** (`MAX_TOOL_ITERATIONS`): the model only *drafts*; whether that draft sends is decided by the two gates, not the model. On escalate the customer sees ONLY a single generic deferral (`GENERIC_DEFERRAL`) - never the gate reason, the retrieval scores, or any access metadata (`_escalated` is the sole constructor of an escalated result, so the non-disclosure invariant is structural). Shipped as a library + scored by the [[E7 escalation eval]], and **wired to a live HTTP endpoint by US-079** - `POST /widget/conversations/messages` runs it as the workspace support bot (`support_bot.run_bot_deflection_turn`, behind the US-077 per-workspace breaker) and streams the result - so its env knobs below are now read by the running backend (the `EscalationConfig` is built + range-validated at backend startup, not just by the eval runner).
- **Retrieval gate** - the cheap, **deterministic** strong/weak verdict that fires first: `strong = (top1_cosine >= τ_sim) AND (n_cleared >= n_min)`, where `top1_cosine` is the max raw cosine across results and `n_cleared` counts rows whose cosine is `>= match_threshold` (`retrieval_gate`, US-047). Both bounds inclusive; empty results - or keyword-only rows carrying no cosine - are weak. Pure arithmetic on scores, no LLM / no I/O, so identical inputs always yield an identical decision (the property that lets it hard-block per-PR). `match_threshold` is **not** a new escalation knob - it is the existing `SEARCH_SIMILARITY_THRESHOLD` (`retrieval.get_similarity_threshold`), reused so "a row cleared retrieval" means the same thing in the gate as in retrieval.
- **Raw cosine surface (US-046)** - every `SearchDocumentsResult` now carries `cosine_similarity`, the pre-fusion vector cosine in a calibrated `[0,1]`, kept **separate from `similarity`** (which RRF fusion and reranking overwrite with the rank-artifact score). Vector rows set it equal to `similarity`; keyword-only rows leave it `None`; fusion and reranking preserve it. This separate field is the calibrated score the retrieval gate thresholds on instead of the RRF rank - the plumbing `backend/test_cosine_surface.py` guards.
- **Faithfulness gate** - the second, LLM-judged gate: **exactly one** structured-output call on the cheap runtime-judge model scoring whether the draft is supported by the retrieved context, escalating below `faithfulness_cutoff` (`faithfulness_gate`, US-048). A net-new RUNTIME gate - NOT the offline multi-call [[RAGAS metric]] `faithfulness` (same word, different machinery). It **fails closed**: a judge error/timeout escalates, never auto-sends. Its model is `JUDGE_MODEL` (`get_judge_model`), default `gpt-4o-mini`, which - unlike the answerer aux-selectors - does **not** chain through `OPENAI_MODEL`, so a big answerer never makes the per-reply gate expensive; the provider/connection is the `judge`-role binding ([[model-surface-doc]], ADR-0006).
- **Escalation config (US-050)** - the three per-request gate knobs, resolved + range-validated once at startup into a frozen `EscalationConfig` (`from_env`): `ESCALATION_TAU_SIM` (τ_sim ∈ [0,1], default `0.4`), `ESCALATION_N_MIN` (≥1, default `2`), `ESCALATION_FAITHFULNESS_CUTOFF` ([0,1], default `0.7`). The defaults are ADR-0003 worked-example placeholders until the E7 sweep knee (US-058) promotes its recommended values. The gates read no env themselves - knobs are passed in - keeping them pure and testable; a fat-fingered knob fails the boot, not the first support request.
- **False-resolve ceiling** - the buyer's single risk-tolerance number: the max fraction of should-escalate questions allowed to auto-resolve, `ESCALATION_FALSE_RESOLVE_CEILING` ∈ [0,1], default **5%** (`get_false_resolve_ceiling`). It is the **pinned SAFETY metric** (the Risk #3 failure: an unanswerable question auto-resolved) - deliberately kept OFF `EscalationConfig` so it can never leak into the per-request latency path, and consumed only by the E7 sweep / knee (US-058) and the E7 CI gate (US-059), never the live pipeline. Distinct from the **tunable quality metrics** (`deflection`, `false-escalate`), whose regressions are governed comment-vs-fail by the configurable E8 gate, not hard-blocked. That security-vs-quality split is now reified as an executable registry - `evals/gate/classes.py` (US-101, Epic F) tags each eval output `security` (pinned `fail`, no loudness knob - querying one raises) vs `quality` (tunable), with `false_resolve` a `quality` metric whose ceiling breach stays pinned (`straddle="ceiling_is_invariant"`). US-102 enforces that classification: the gate-declaration loader (`evals/gate/declaration.py`) hard-rejects any `off|comment|fail` verdict on a `security` output ("delete the eval to remove it"), and security invariants are binary asserts (`evals/gate/security.py:check_no_access_zero_leak` pins `security_no_access[filter][mode] == 1.0`, failing the run non-zero on a leak) - so silencing one requires *deleting* its eval, never a config flag. US-103 extends that same declaration with a `bindings:` section - the RAGAS gates' project bindings (cells, thresholds, cross-family judge map) that `evals/retrieval/ragas_gates.py` used to hardcode, lifted into a `GateBindings` config object the detection functions take and shipped as `evals/gate/gate.yaml`, which reproduces today's constants byte-for-byte (omitting the `corroboration:` sub-block degrades a RAGAS drop to `single-judge-red`). US-104 adds the tunable `off|comment|fail` verdict-to-action layer over the quality metrics (`evals/gate/verdict.py`): one loudness knob per quality suite (`retrieval_metrics`/`ragas`/`escalation`, which partition every quality output) mapping `(severity, knob) → action` (`fail` ⇒ red blocks / yellow comments, `comment` ⇒ both comment, `off` ⇒ silent, default `comment`), with a `false_resolve` ceiling breach blocking regardless of the knob and a per-output `verdicts:` override winning over its suite. US-105 adds the orthogonal **placement** axis (`evals/gate/placement.py`): a `per_pr:` section opts *deterministic* quality gates into per-PR merge-blocking and structurally rejects a per-PR `fail` on a non-deterministic (LLM-judged) gate (`PlacementError`, a load-time error) - so determinism, not buyer preference, decides what may red-bar a merge, with `GateDeclaration.blocks_merge` / `files_issue` the consuming seam still not wired onto the runner-exit / PR-comment call sites.
- **E7 escalation eval** - the deflection-pipeline golden set (`evals/retrieval/escalation_gold.yaml`) + runner (`evals.retrieval.e7_runner`) over four populations - **P1a** (no context → must escalate at the retrieval gate), **P1b** (a P2 replayed under a no-access viewer → must escalate AND emit the byte-identical generic deferral, US-058), **P2** (answerable + faithful → must auto-resolve; an escalation is a *false-escalate* annoyance), **P3** (the moat: strong retrieval but no faithful answer → must draft then escalate at the faithfulness gate; an auto-resolve is the *false-resolve* safety failure). The legs split on **determinism**, which decides CI placement (US-059): the **deterministic** P1a/P1b retrieval-gate + non-disclosure legs are the per-PR tripwire that may hard-block a merge (no LLM, no `ANTHROPIC_API_KEY`); the **LLM-judged** P2/P3 legs (offline cross-family Claude judge) + knob sweep run **weekly**, so a faithfulness-leg false-resolve carries an accepted up-to-a-week detection latency. Full treatment: `docs/evals.md` § 6.

## Permissions eval design (Module 11)

- **Suite shape**: 50 golden questions × 3 viewer setups = 150 runs per eval mode. The three viewer setups are *Full-access*, *Partial-access*, and *No-access*; each is a deterministic function of the question's gold chunks (defined in the YAML, not hand-picked per question).
- **Three independent claims, three tables**: Security (no-access viewer should retrieve zero gold under both modes), Recall trade-off (partial-access viewer is where post-filter recall collapses and pre-filter recovers — the headline number), Non-regression (full-access viewer matches the unfiltered Module-10 baseline within tolerance).
- **Selectivity policy**: For the 14-chunk *correctness eval*, partial-access = "gold chunks + N random others" with N fixed. For the 10k-chunk *scale benchmark*, partial-access = "gold chunks + k% of filler" for k ∈ {50, 10, 1}.

## Frontend share UX (Module 11)

- **Scope (single dialog, ~80 lines of React using existing shadcn Dialog + Badge):**
  - One input + "Grant" button. Accepts email (resolves to user) or group name (resolves to group). Free-text — **no autocomplete**.
  - List of current grants, one row per principal, "x" button revokes.
  - Owner row, always present, never removable. Display-only — the owner's access comes from `chunks.user_id`, not from a `chunk_acl` row.
  - Disabled while the document's `status != 'ready'`.
- **Retrieval-side reveal (the load-bearing UI):** existing tool-attribution panel (Module 7/8 `ToolAttribution`) gains a per-chunk badge showing which principal granted access: "via engineering" or "via direct grant." This is what makes the demo convincing — without it, the demo is "watch results change"; with it, the demo is "watch *why* the results changed."
- **Three API endpoints**, all gated on `documents.user_id = auth.uid()` (only the doc owner can share):
  - `POST   /api/documents/{id}/share` — body `{ principal_email_or_name }` → resolves, then inserts one `chunk_acl` row per chunk in a transaction. Idempotent.
  - `DELETE /api/documents/{id}/share/{principal_id}` — deletes all `chunk_acl` rows for the doc × principal.
  - `GET    /api/documents/{id}/shares` — aggregates `chunk_acl` over the doc's chunks to render the dialog.
- **Two columns added to `chunk_acl`** beyond the (chunk_id, principal_type, principal_id) primary key: `granted_by uuid references auth.users(id)` and `created_at timestamptz default now()`. No UI for either in v0; they exist so a future audit-log view is possible without a backfill.
- **Explicit non-goals in v0** (call out each in the writeup with a one-line reason): per-chunk override UI, autocomplete, bulk operations, "what would Sarah see" preview, role/permission distinction (binary grant only), audit-log UI.
- **Edge cases:**
  - Non-existent email → backend returns 4xx, frontend toasts "No user with that email. They have to sign up first." No auto-create.
  - Same principal granted twice → idempotent no-op, no error.
  - Owner cannot be revoked → revoke button absent on owner row.
  - Doc still ingesting → share button disabled.

## Share-UX implementation concerns (still open — pin before build)

- **Email-to-user UUID resolution.** `auth.users` is not directly queryable by authenticated callers without service-role. Three options: (i) create `public.profiles(id, email)` mirrored from `auth.users` via a trigger and an RLS policy allowing authenticated reads, (ii) use the Supabase admin SDK from the backend with service-role, (iii) write a SECURITY DEFINER RPC `lookup_user_by_email(text) returns uuid` that returns NULL for non-existent emails. Recommend (i) — cleanest reuse, no new service-role surface in the backend. Pin during US authoring.
- **`match_chunks` must return the granting principal per row** to populate the badge. The current `EXISTS (...)` predicate doesn't surface this. Need to restructure as a join with `DISTINCT ON (c.id)` picking a representative principal per chunk (precedence: direct user grant > group grant; tiebreak deterministic). Non-trivial SQL change — flag in the SQL diff in the writeup.
- **Group-name registry.** v0 spec accepts "group name" as the share input, but there's no `principals` table named yet. Either (a) add a `principals(id uuid, name text unique, kind text default 'group')` table that the seed script populates, or (b) treat groups as having UUIDs only and the share dialog requires the group UUID (worse UX). Recommend (a). Pin during US authoring.
- **Doc-owner authorization on the share API.** The three endpoints check `documents.user_id = auth.uid()` server-side (the share API has its own permission model — only owners can share). Worth a line in the writeup; this is the kind of thing reviewers look for.

## Rollout (Module 11)

- **No feature flag.** Module 11 is a correctness migration, not a tool-availability gate. Owner-OR-ACL falls back to owner-only when `chunk_acl` is empty, so existing single-user behavior is preserved without a flag. Convention in this repo is that flags gate *tool availability* (e.g., `crm_tool_enabled`), not data-path behavior.
- **No backfill.** Empty `chunk_acl` rows for existing chunks + the additive predicate = owner-only behavior, identical to today.
- **Three migrations**: `principal_membership`, `chunk_acl`, `match_chunks` replacement. The load-bearing perf index is `chunk_acl(principal_id, chunk_id)` (principal-leading so the EXISTS subquery is index-served).
- **CI split**: per-PR runs the **correctness eval** (50 × 3 viewer setups = 150 runs). The **scale benchmark** (10k Wikipedia filler + `ef_search` sweep) runs **nightly + manual `workflow_dispatch`**, never per-PR. Nightly workflow must fail loudly (notification) on threshold breach, since nobody reads nightly artifacts unless something pages them.

## HNSW + selective-filter mitigation (Module 11)

- **Demonstrated lever**: `ef_search` tuning. `match_chunks` gains an optional `ef_search int` parameter; when set, the function calls `perform set_config('hnsw.ef_search', ef_search::text, true)` before the SELECT. The scale benchmark sweeps it across a fixed grid (e.g., 40 / 80 / 200 / 500) per selectivity bucket and writes the recall-vs-`ef_search` curve.
- **Discussed only (not implemented in v0)**: partial indexes per principal group (only useful if some groups are hot — has real operational cost and doesn't generalize); IVFFlat (lower unfiltered ceiling, better filtered behavior — a trade-off, not a free upgrade). Justify both in the writeup; do not ship.

## Evals (Module 10 + Module 11)

- **Correctness eval** — the existing 50-question golden set against the 7-doc / 14-chunk corpus in `db_seed/corpus/`. Demonstrates *correctness* dynamics (pre-filter vs post-filter recall under sparse permissions). Stays in PR CI.
- **Scale benchmark** — a separate synthetic harness on ~10k Wikipedia-derived chunks. Demonstrates *HNSW × selective-filter* dynamics (`ef_search` tuning, IVFFlat comparison). Runs out of PR CI — nightly or manual `workflow_dispatch`. The 10k size is chosen as ~10× pgvector's default `ef_search`=40; below that, the HNSW walk is essentially exhaustive and the phenomenon is invisible.
- **Filler corpus** — the Wikipedia-derived chunks exist to populate the HNSW neighborhood, NOT to be answered about. Golden questions remain anchored to the original 7 docs. The writeup must be explicit that filler chunks are never gold.
- **RAGAS metric** — one of the four canonical RAG-evaluation scores (`faithfulness`, `answer_relevancy`, `context_precision`, `context_recall`) on a **0–1** scale, computed by the `ragas` library with `gpt-4o-mini` as judge. Distinct from the custom Claude-judge scores (`faithfulness`, `helpfulness`) on a **1–5** Likert scale — same English word "faithfulness", two different numbers, so always state the scale. RAGAS ships *alongside* the Claude judge, never as a replacement.
- **Same-family bias** — the RAGAS judge (`gpt-4o-mini`) shares a model family with the answer generator (also `gpt-4o-mini`), so it can be systematically lenient toward outputs that "reason like it does." Accepted deliberately: `gpt-4o-mini` is cheap enough to score four metrics weekly, and judge independence is preserved by the *cross-family* Claude judge, which stays the headline signal.
- **Cell** — a (viewer × filter) tuple, e.g. `full_access:pre_filter`. The retrieval runner sweeps six; RAGAS scores only the two `pre_filter` cells (`full_access`, `partial_access`) — the other four are degenerate or already covered by the security / recall trade-off tables.
- **Effective coverage** — per (RAGAS metric × cell), the fraction of questions that produced a **non-NaN** score (`non-NaN / total`). Distinct from "tried": a metric can be attempted on all 50 questions yet have 0.92 coverage because the judge refused or the contexts were empty on 4 of them. The headline `mean_strict` divides by *total* (NaN→0), never by the non-NaN count, so degraded coverage cannot hide behind a shrinking denominator.
- **Cross-family corroboration** — the alert rule: a RAGAS score drop escalates to a **red** regression only when the independent cross-family Claude judge shows the same drop in the same [[Cell]]. An uncorroborated single-judge drop stays **yellow**. Two independent observations agreeing is a far stronger signal than one — the rule suppresses single-judge false alarms.
- **single-judge-red** — the finding tag for a Context Precision / Context Recall regression. Those two metrics have no Claude-judge equivalent to corroborate against, so a drop fires red on the RAGAS judge alone — flagged `single-judge-red` so a reader knows it rests on one judge, and given a 2-week auto-close window (vs 1 week for cross-family-corroborated reds) because there is no second judge to clear it sooner.
